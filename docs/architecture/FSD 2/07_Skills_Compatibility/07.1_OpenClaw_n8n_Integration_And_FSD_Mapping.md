# OpenClaw × MAXshot（n8n）集成与 FSD 映射报告

> **日期**：2026-02-02  
> **FSD 位置**：`07_Skills_Compatibility/`（唯一存放）  
> **面向**：LEO（产品决策）/ Cat（技术边界裁决）/ Alex（n8n 方案落地）  
> **目标**：判断能否用 OpenClaw 作为"入口与控制平面"，把底层通信/会话/多端接入交给 OpenClaw，同时保持 MAXshot 的确定性编排（n8n + Supabase）不被破坏。

---

## 1. 结论摘要（可直接用于决策）

- **结论 1（可行性）**：可行。OpenClaw 具备两类关键能力：
  - **HTTP Webhooks（入站触发）**：`POST /hooks/*` 可把外部事件注入 OpenClaw（wake / agent / mapping）。
  - **HTTP Tools Invoke（无 LLM 的工具直调）**：`POST /tools/invoke` 可在不运行"完整 agent 回合"的情况下，直接调用 OpenClaw 的工具（例如 `message` 发送 Telegram/Slack/Discord 消息）。这对 n8n 集成是关键。

- **结论 2（推荐定位）**：OpenClaw 应定位为 **"入口网关 + 控制面（工具/Skill 注册、权限与审计、回调通知）"**；
  **确定性编排与业务状态机仍以 n8n + Supabase 为主**。

- **结论 3（风险点）**：最大风险是把 Router/状态机迁入 OpenClaw 的 LLM 回合会破坏既有宪法（Router 确定性、可审计；Intent Analyzer 禁止输出 `capability_chain`）。

## 2. OpenClaw 代码库证据（与我们最相关的能力）

### 2.1 Webhooks：OpenClaw 可被外部系统触发

- 配置入口：`~/.openclaw/openclaw.json` 的 `hooks` 字段
- 文档：`docs/automation/webhook.md`、`docs/gateway/configuration.md`
- 行为：
  - `POST /hooks/wake`：给 **main session** 注入系统事件
  - `POST /hooks/agent`：运行一个 **隔离的 agent turn**（可选 deliver 回消息渠道）
  - `POST /hooks/<name>`：通过 `hooks.mappings` 做 payload→action 的映射与可选 transform

> 适用场景：n8n 执行完成后回调 OpenClaw，触发"通知/总结/人工确认"等。

### 2.2 Tools Invoke（HTTP）：n8n 可"直调 OpenClaw 工具"，无需 LLM

- 端点：`POST /tools/invoke`（与 Gateway 同端口）
- 文档：`docs/gateway/tools-invoke-http-api.md`
- 实现：`src/gateway/tools-invoke-http.ts`
- 关键点：
  - 受 **Gateway auth** + **tool policy** 双重约束（可做 allowlist）
  - 允许指定 `sessionKey`，并可用 header 传递 `x-openclaw-message-channel` 等辅助策略解析

> 这意味着：我们可以把 OpenClaw 当成"统一消息发送/渠道动作执行器"，让 n8n 通过 HTTP 调用 OpenClaw 的 `message` 工具来回消息、发按钮、管理 thread 等。

### 2.3 `message` 工具：统一对外发消息能力

- 工具名：`message`
- 关键动作：`action: "send"`（以及 react/poll/edit/delete/thread 等）
- 实现：`src/agents/tools/message-tool.ts`
- CLI 参考：`docs/cli/message.md`

> 结合 `POST /tools/invoke`：n8n 可以在任意流程节点直接发 Telegram/Slack/Discord/WhatsApp 等回复，不需要自己维护各平台 SDK。

### 2.4 Skills 配置：可加载自定义 Skill 包

- 配置：`skills.load.extraDirs` + `skills.entries.*`
- 文档：`docs/tools/skills-config.md`

> 适用场景：在 OpenClaw 侧提供一个"n8n 触发 skill"（比如 `maxshot-n8n`），由 OpenClaw 统一把用户输入转为结构化 payload 并调用 n8n webhook/API。

## 3. n8n Workflow 如何通过"配置"纳入 OpenClaw（推荐三种路径）

### 路径 A（推荐，最稳）：n8n → OpenClaw（工具直调）做"统一对外输出"

**用途**：让 OpenClaw 承担"多平台消息/动作执行"，n8n 承担"确定性编排"。

- n8n 侧：在工作流的输出节点调用
  - `POST /tools/invoke`，tool=`message`，action=`send`
- OpenClaw 侧：
  - 配置 `gateway.auth`（token/password）
  - tool policy allowlist：只开放 `message`（以及必要的 `sessions_*` / `gateway` 工具，按需）

**优点**：
- 不需要在 n8n 为每个平台维护 SDK/权限与差异化参数
- 不需要 LLM 参与即可发送消息（更符合"确定性/可审计"要求）

**注意**：
- `target` 的格式必须按 OpenClaw 各 channel 规范（见 `docs/cli/message.md`）


### 路径 B（低开发）：OpenClaw → n8n（Skill 触发）做"统一入口转发"

**用途**：用户从 Telegram/WhatsApp 等进入 OpenClaw；OpenClaw 通过一个自定义 Skill 调用 n8n 的 Webhook 来启动对应工作流。

- OpenClaw 侧：增加一个自定义 Skill 目录（例如 MAXshot 仓库内的 skill pack）并在 `skills.load.extraDirs` 指向它
- Skill 的逻辑：用 `curl`/node/python 调用 n8n webhook，并把 OpenClaw 消息上下文（channel/from/sessionKey）作为 payload 发过去

**优点**：
- 入口统一、配置化，接入成本低

**缺点**：
- 入口处仍会经过一次 OpenClaw 的"agent 决策"（由系统 prompt 决定是否调用该 Skill）；需要用 prompt/工具策略强约束"raw_query 必须转发"。


### 路径 C（需要开发，但最干净）：OpenClaw 插件/Hook 自动转发 inbound 到 n8n

**用途**：完全跳过 LLM，在 OpenClaw 收到消息时就把原始事件转发给 n8n。

- 方向：用 OpenClaw 的 plugin hook（例如 `message_received`）在收到消息时调用 n8n webhook
- 产物：一个 OpenClaw plugin（比 Skill 更偏系统级）

**优点**：
- 真正做到"底层通信交给 OpenClaw，但编排完全由 n8n 决定"

**缺点**：
- 需要写 OpenClaw 插件代码（但仍远低于从头做整套 gateway）

## 4. FSD → OpenClaw 配置与 Skills 的映射

### 4.1 输入边界（来自 FSD 06.1 Intent Analyzer）

FSD 要点（摘录）：
- raw_query（Telegram/Chat 文本）→ 必走 Intent Analyzer
- structured / timeline 输入 → 跳过 Intent Analyzer
- Intent Analyzer 输出契约：只允许 `intent` + `extracted_slots`；严禁 `capability_chain`

**映射到 OpenClaw**：
- **渠道输入**：OpenClaw `channels.telegram` / `channels.webchat` 负责接入
- **会话维持**：OpenClaw `session.*` 负责"对话线程与历史"（但业务状态仍在 Supabase）
- **Intent Analyzer 执行位置**：建议仍在 n8n（可审计、可替换），OpenClaw 只做入口与转发

落地建议：
- 若走"路径 B"：在 OpenClaw 的系统提示中写死规则：所有 raw_query 必须调用 `maxshot-n8n.intent_analyzer`（Skill），并把结果写回 n8n 的 `task_executions.payload`。
- 若走"路径 C"：由插件/Hook 直接把 raw_query 转发给 n8n 的 Intent Analyzer workflow。

### 4.2 发布系统（来自 Master Publisher System · FSD）

FSD 要点（摘录）：
- `publishing_timeline` 是统一排期与状态机（scheduled→publishing→published/failed）
- `18_Master_Publisher_Router` 定时扫描 + 防重
- Thread 路径复用 `13_Thread_Post_deploy` 黑盒

**映射到 OpenClaw**：
- OpenClaw 不承担发布状态机；它只承担：
  - **人工入口**：例如 Telegram 下发"立即发布/重试/取消"等指令
  - **结果通知**：n8n 发布成功/失败后，调用 OpenClaw `POST /tools/invoke` → `message.send` 发回用户/群

落地建议（最小改造）：
- n8n 中保持 `18` 的定时扫描（与 FSD 一致）
- 在 `06_Single_Post_Executor_v2` 与 `13_Thread_Post_deploy` 的结果节点：
  - 同步回写 Supabase 状态
  - 再调用 OpenClaw `/tools/invoke` 发送通知

## 5. 推荐的 MVP（最小可行闭环）

### 5.1 MVP 目标

跑通：
- **Telegram 输入** →（raw_query）→ Intent Analyzer（n8n）→ 写 `task_executions.payload`
- （后续）写入 `publishing_timeline`
- `18_Master_Publisher_Router` 调度 → `06_Single_Post_Executor_v2` 发布
- 发布完成回写状态 → **通过 OpenClaw 统一回消息**（/tools/invoke → message.send）

### 5.2 MVP 为什么选这个路径

- 完全遵守 Intent Analyzer 的 guardrails
- 保留 Master Publisher 的状态机与防重逻辑
- 让 OpenClaw 真正承担"底层渠道输出"，并且不依赖 LLM（更稳）

## 6. 配置要点清单（给 Alex/Cat 的落地 checklist）

### 6.1 OpenClaw 侧（配置项）

- `gateway.auth.*`：启用 token 或 password（n8n 调用 /tools/invoke 需要）
- `tools.allow`：至少 allow `message`（建议先最小化 allowlist）
- `channels.telegram.*`：allowFrom、groupPolicy 等（入口安全收口）
- （可选）`hooks.*`：用于 n8n 回调触发（若需要 wake/agent/mapping）

### 6.2 n8n 侧（节点方式）

- 用 HTTP Request 节点调用 OpenClaw：
  - `POST http://127.0.0.1:18789/tools/invoke`
  - header：`Authorization: Bearer <gateway token>`
  - body：tool=`message` / action=`send` / args={channel,target,message}
