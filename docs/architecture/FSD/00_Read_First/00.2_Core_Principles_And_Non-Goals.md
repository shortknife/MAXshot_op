# 00.2 核心原则与 Non-Goals

> **用途**：明确「我们要什么」「我们不要什么」，防止工程师擅自补智能

---

## 一、核心原则

### 1.1 宁可多问，不可越权
MAXshot 遇到 Grey Area（未被 Allowed/Forbidden 明确覆盖）时，**默认 Require Confirmation**。  
不擅自越权执行。

### 1.2 不留痕 = Bug
所有决策、拒绝、确认、执行，必须写入审计。  
无 trace = 不可信 = Bug。

### 1.3 Execution 是琥珀
Execution 一旦创建，其上下文在启动时冻结为 Snapshot。  
禁止「偷懒 join 最新表」改写历史。

### 1.4 拒绝路径是安全核心
Entry Reject、Capability Reject、Soul Reject 有明确路径与话术。  
不凭空 throw error，不模糊拒绝。

### 1.5 Soul 是滤镜，不是组件
Soul 永远不被「调用」。  
Soul 只在「选择」发生时起作用，无 Runtime API。

---

## 二、Non-Goals（禁止工程师擅自添加）

| 禁止项 | 说明 |
|--------|------|
| LLM 直接调度 | Router 必须由 Code 实现，LLM 只做 Intent Analysis |
| 模型自判风险等级 | 风险等级只能来自 Policy / Code |
| random / average 决策 | 冲突裁决有明确优先级：Soul > Insight > Experience |
| 联表查最新状态 | Execution 启动时 Snapshot 冻结，不 join 实时表 |
| Soul 作为 Service | Soul 无 API，无 mock，无独立调用 |
| 在线黑箱学习 | Evolution 对 Router 只能通过 Recommendation（可审计）影响 |
| 用户记忆演化 | session_context 不跨 Turn 隐式演化，不承担长期记忆 |
