## 1. 定义与边界

**Session Context Snapshot（简称 session_context）** 是在调用 Intent Analyzer 之前，由 Context Load 生成的一次性快照，用于：

- 支持对话连续性（如指代消解："那 USDT 呢？"）。
- 提供与当前用户/会话相关的 minimal 上下文（最近任务、最近问题等）。

**重要边界**：

- session_context **不跨 turn 隐式演化**，每次调用 `/context/load` 时重新生成；
- session_context **不同于 memory_refs**：后者是 Router 在任务分解时用于 Capability 的知识引用；
- session_context 仅供 Intent Analyzer 使用，不直接传入 Capability 层。

---

## 2. 顶层结构

```json
{
  "user": {
    "user_id": "string",
    "preferred_language": "string | null"
  },
  "recent_tasks": [
    {
      "task_id": "string",
      "intent": "string",
      "slots": {},
      "created_at": "ISO8601 timestamp"
    }
  ],
  "recent_messages": [
    {
      "message_id": "string",
      "text": "string",
      "timestamp": "ISO8601 timestamp"
    }
  ],
  "derived_state": {
    "last_intent": "string | null",
    "last_slots": {},
    "last_metric": "string | null"
  }
}
```

### 字段说明

- **user**: 当前会话归属的用户信息。
- **recent_tasks**: 与本会话相关的最近 N 条任务摘要（例如最近 5 条），用于从历史任务中继承 intent/slots。
- **recent_messages**: 最近 N 条对话消息（如最近 10 条），用于辅助 LLM 理解上下文语境。
- **derived_state**: Context Load 基于 recent_tasks 与 recent_messages 计算出的「快捷状态」，避免每次都让 LLM自己推断。

---

## 3. 生成流程 (由 Context Load 实现)

1. 根据 `requester_id` / `chat_id` 查询最近会话相关的 tasks 与 messages；
2. 选取最近 N 条任务与消息，并按时间排序；
3. 计算 derived_state：
   - last_intent: 最近一次成功任务的 intent；
   - last_slots: 最近一次任务中核心 slots；
   - last_metric: 若上一次是 ops_query，抽取其中的 metric；
4. 将上述数据组装为 session_context，并注入到 EntryRequest.payload.session_context。

---

## 4. 与 Intent Analyzer 的协作方式

### 4.1 在 Prompt 中的使用

在 Intent Analyzer 的 User Prompt Template 中，session_context 将以 JSON 形式注入：

```json
{
  "user_query": "{{clean_query}}",
  "session_context": {{session_context}},
  "intent_schema": "{{intent_schema}}"
}
```

Intent Analyzer 需遵守：

- 优先使用 derived_state 中的信息进行指代消解（如 last_intent, last_metric）；
- 当用户问题与历史任务冲突时，以**用户最新输入**为准；
- 若 session_context 为空或不完整，应安全退化为「单轮解析」。

---

## 5. 不变性与约束

1. session_context **只读**：Intent Analyzer 不得修改 session_context 的内容，只能基于它推断 intent/slots；
2. Context Load 是 session_context 的唯一写入者；
3. Router 与 Capability 不直接依赖 session_context，避免强耦合；
4. session_context 的结构一旦在 FSD 冻结，后续只允许向下兼容扩展（新增字段），不允许删字段或变更语义。
