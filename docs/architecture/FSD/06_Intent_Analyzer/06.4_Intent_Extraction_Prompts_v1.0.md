## 一、设计原则

### 1.1 四层架构原则（强制）

所有 Intent Extraction Prompt 必须遵循 MAXshot 四层 Prompt 架构规范：

| Layer | 说明 | Token 预算 | 职责 |
|-------|------|-----------|------|
| **Layer 1: Core Persona** | 定义角色本质 | 200-300 tokens | "你是谁" - Intent Analyzer 作为 Orchestration Layer 的网关 |
| **Layer 2: Style & Voice** | 定义输出风格 | 100-150 tokens | "怎么输出" - 分析性、直接、结构化 |
| **Layer 3: Behavior Rules** | 定义工作逻辑 | 100-200 tokens | "做什么" - 意图分类、槽位归一化、上下文继承 |
| **Layer 4: Dynamic Context** | 注入场景信息 | 200-400 tokens | "在什么背景下" - Memory Layer、Session Context |

### 1.2 外置存储原则（禁止硬编码）

- ❌ **禁止硬编码业务 Prompt 到 n8n Code 节点**
- ✅ **所有业务 Prompt 必须存储在 Supabase `prompt_library` 表**
- ✅ **n8n Code 节点只负责变量注入和模板化**

### 1.3 测试驱动原则（强制验证）

- ✅ 必须提供测试用例 JSON（输入数据示例 / 预期输出示例 / 测试场景说明）
- ❌ 禁止未测试直接上线

## 二、Intent Analyzer v2 Prompt 设计（参考实现）

### 2.1 System Prompt（四层逻辑架构）

```markdown
## Layer 1: Core Persona Layer
You are the **MAXshot Intent Analyzer**. As the gateway to the Orchestration Layer, your mission is to decode user natural language into structured execution parameters. You represent the system's "Analytical Brain" that understands DeFi context and user needs.

## Layer 2: Style & Voice Layer
- **Analytical & Direct**: Do not use conversational filler.
- **Structural Integrity**: Your response must be a single, valid JSON object.
- **Language**: Use English keys; preserve original semantic value in string fields.

## Layer 3: Behavior Rules Layer
1. **Intent Classification**:
   - Categorize into: `ops_query`, `marketing_gen`, `documentation`, `mixed`, or `unknown`.
2. **Slot Normalization**:
   - `vault_name`: Resolve keywords to full names (e.g., "USDC" -> "USDC Vault").
   - `metric`: Strict extraction from [`APY`, `TVL`, `Allocation`, `Rebalance_Reason`].
3. **Context Persistence**:
   - Use `{{session_context}}` to understand previous state and elliptical queries.
   - If user asks an elliptical question ("What about USDT?"), inherit the `metric` from the previous task in context.
4. **CONSTITUTIONAL FORBIDDEN ACTIONS**:
   - **DO NOT** output `capability_chain`.
   - **DO NOT** suggest specific tools or workflows.
   - **DO NOT** generate prose or explanations.

## Layer 4: Dynamic Context Layer
{memory_layer_context}
```

### 2.2 User Prompt Template（输入契约）

```json
{
  "user_query": "{{clean_query}}",
  "session_context": "{{session_context}}",
  "intent_schema": "{{intent_schema}}",
  "vault_list": ["dForce USDC - Morpho - arbitrum", "dForce USDT - Morpho - arbitrum"]
}
```

### 2.3 Output Envelope（输出契约）

```json
{
  "intent": "string",
  "extracted_slots": {
    "vault_name": "string | null",
    "metric": "string | null",
    "time_range": "string | null",
    "is_fuzzy": "boolean"
  },
  "context_inherited": "boolean"
}
```

### 2.4 `memory_layer_context` 语义与格式约定

**产品定义**（LEO 产品定义 §五）：`memory_layer_context` 是给 **Intent Analyzer LLM 的「开卷」上下文**，用于约束 LLM 只输出合法 intent、减少 Gate 驳回，并帮助理解用户话术。

| 内容 | 是否必须 | 说明 |
|------|----------|------|
| **合法 intent/capability 简要描述** | ✅ 必须 | 名称 + 一句话说明，让 LLM 只输出合法 intent；无此则 Gate 频繁驳回。建议格式：`intent_name` + `one_line_description`（权威来源：Lucy 的 Skill/intent 清单）。 |
| **产品/运营共识摘要** | 建议 P1 | product_summary、core_concepts、风格指南等，帮助理解用户话术。 |
| **与意图相关的 capability 子集** | 可选 | db_schema 摘要、品牌词表等；按 token 预算与需求逐步扩展。 |

**与 Entry Memory / session_context 的边界**（同源不同形）：

- **Entry Memory**：Gate 判定用的权威完整列表（合法 intent/capability、规则）。
- **session_context**：对话连续性、指代消解，一次快照。
- **memory_layer_context**：给 LLM 的裁剪摘要；可与 Entry Memory 同源（如均来自 Lucy 的 intent 清单），Gate 用完整列表校验，LLM 用简短描述约束输出。

**首版硬性要求**（LEO §五）：至少包含「合法 intent 一句话描述」，保证 LLM 输出在合法集合内。  
**职责归属**：由 SAM Context Load 扩展输出 `memory_layer_context`（或 n8n 过渡拉取并 Merge）；Alex 在工作流中替换 `{{memory_layer_context}}` 占位符。

## 三、Prompt 变量命名规范

### 3.1 输入变量命名

| 变量名 | 类型 | 说明 | 示例 |
|-------|------|------|------|
| `clean_query` | string | 清洗后的用户查询 | "How much is the TVL for USDC?" |
| `session_context` | object | 会话上下文快照 | 包含前几轮对话的状态 |
| `intent_schema` | object | 意图 Schema 定义 | 合法意图列表和约束 |
| `memory_layer_context` | string | Memory Layer 上下文 | 合法 intent 描述 + 产品共识摘要 |
| `vault_list` | array | 合法金库名称列表 | ["USDC Vault", "USDT Vault"] |

### 3.2 输出字段命名

| 字段名 | 类型 | 说明 |
|-------|------|------|
| `intent` | string | 意图分类（`ops_query`, `marketing_gen`, `documentation`, `mixed`, `unknown`） |
| `extracted_slots` | object | 提取的槽位信息 |
| `extracted_slots.vault_name` | string \| null | 金库名称 |
| `extracted_slots.metric` | string \| null | 指标类型 |
| `extracted_slots.time_range` | string \| null | 时间范围 |
| `extracted_slots.is_fuzzy` | boolean | 是否模糊匹配 |
| `context_inherited` | boolean | 是否继承上下文 |

## 六、与 n8n 集成的实现模式

### 6.1 Code 节点职责

Code 节点不应该包含 Prompt 文本，只负责：

1. **从 Supabase 读取 Prompt**：
   ```javascript
   // 从 prompt_library 表读取 intent_analyzer_v2
   const prompt = await supabase
       .from('prompt_library')
       .select('*')
       .eq('slug', 'intent_analyzer_v2')
       .single();
   ```

2. **变量注入和模板化**：
   ```javascript
   // 替换 System Prompt 中的占位符
   const systemPrompt = prompt.system_prompt
       .replace('{memory_layer_context}', memoryLayerContext);

   // 替换 User Prompt Template 中的占位符
   const userPrompt = JSON.stringify(JSON.parse(prompt.user_prompt_template)
       .replace('{{clean_query}}', cleanQuery)
       .replace('{{session_context}}', JSON.stringify(sessionContext))
       .replace('{{vault_list}}', JSON.stringify(vaultList)));
   ```

3. **调用 LLM API**：
   ```javascript
   // 使用注入后的 Prompt 调用 Claude/OpenAI API
   const response = await anthropic.messages.create({
       model: JSON.parse(prompt.model_config).model,
       system: systemPrompt,
       messages: [
           { role: 'user', content: userPrompt }
       ],
       temperature: JSON.parse(prompt.model_config).temperature
   });
   ```

### 6.2 禁止模式

❌ **禁止在 Code 节点中硬编码 Prompt 文本**：

```javascript
// ❌ 错误示例：硬编码 Prompt
const systemPrompt = `You are the MAXshot Intent Analyzer... (完整 Prompt 文本)`;

// ✅ 正确示例：从 Supabase 读取
const systemPrompt = (await supabase.from('prompt_library').select('system_prompt')
    .eq('slug', 'intent_analyzer_v2').single()).data.system_prompt;
```
