# 04.2 Memory 类型与角色

> **用途**：Foundation / Experience / Insight 的职责与来源

---

## 一、Memory 原子类型

| 类型 | 定义 | 来源 |
|------|------|------|
| **Foundation** | 业务公理、硬规则 | 产品/运营显式维护 |
| **Experience** | 经量化验证的成功/失败路径 | 历史 Execution 的 Outcome |
| **Insight** | 高维模式（如「熊市用户偏好黑色幽默」） | Evolution Engine 的 Self-Reflection |

---

## 二、权重

- Foundation：1.0（不可变）
- Experience：随验证次数动态调整（**v5.1 约束**：权重仅用于 Sandbox/离线分析，**运行时不做动态权重推理**，为 v6.0 兼容）
- Insight：高权重，进化结晶（v5.1 Phase 3 后启用）

---

## 三、Memory 闭环架构留位（MVP 架构补全）⭐

> **目的**：在架构层预留 Memory 闭环的注入点与契约，实现可放在 Phase 1.5/2。

| 留位项 | 说明 | 实现阶段 |
|--------|------|----------|
| **memory_refs 注入点** | Router 任务分解输出 `memory_refs`；组装 CapabilityInput 时写入 `context.memory_refs`，Capability 只读使用，不直接访问 DB。 | 已定义（Integration Spec 2.2）；注入逻辑 Phase 1.5/2 |
| **Experience 写入契约** | Capability 执行完成后，符合条件的结果可写入 Experience（如成功路径、用户反馈）；写入格式、触发条件由 Evolution / Policy 定义。 | Phase 1.5/2 |
| **检索注入时机** | 下次请求时，Sam `/context/load` 或等价逻辑根据 execution/requester 检索 Foundation + Experience + Insight，生成 memory_layer_context，注入 Intent Analyzer 或 Router。 | Phase 1.5/2 |

**约束**：Memory 闭环不阻塞 Phase 1 主链路验收；架构文档与 FSD 先写清留位，便于实现时对齐。
