# 01.2 用户路径：Rejection Paths ⭐

> **用途**：System Safety 核心文档。定义三类拒绝与行为差异。

---

## 一、三种拒绝类型

| 类型 | 触发点 | 含义 | 系统行为 |
|------|--------|------|----------|
| **Entry Reject** | Entry Gate | 不在职责范围，不接单 | 不创建 Task/Execution，返回拒绝话术 |
| **Capability Reject** | Capability 执行 | 无法胜任（如缺参数、超边界） | 创建 Execution，status=failed，写入 audit |
| **Soul Reject** | Entry/Intent/Content | 价值/安全/冒犯，不允许输出 | 不进入执行态，返回 Soul 拒绝话术 |

---

## 二、Entry Reject

**触发条件**：
- 明确违法/风险/超边界
- 不在合法 intent/capability 集合内

**行为**：
- 不创建 Task
- 不创建 Execution
- 返回 Entry Reject 话术（见 01.4）

**审计**：记录 Reject 原因、timestamp、entry_type（用于事后分析）

---

## 三、Capability Reject

**触发条件**：
- Capability 执行失败（缺参数、超时、外部服务异常）
- 不满足 Capability 的 Policy Gate

**行为**：
- 已创建 Execution
- 更新 status=failed
- 写入 audit（error、failure_mode）

**与 Entry Reject 区别**：Capability Reject 表示「系统尝试了，但能力不足以完成」。

---

## 四、Soul Reject

**触发条件**：
- raw_query 触及 Soul 边界（如：不制造焦虑、不欺骗）
- 意图或内容不符合 MAXshot 人设

**行为**：
- 不进入执行态
- 返回 Soul 拒绝话术（见 01.4）

**审计**：记录 Soul Reject 原因、timestamp

---

## 五、Continue Chat ≠ Reject

| 场景 | 行为 |
|------|------|
| 信息不足、意图模糊 | Continue Chat（澄清，最多 3 轮） |
| 达到轮次上限仍无法确定 | status=unknown 或转人工 |

**原则**：除明确风险外，不 Reject 模糊输入；先澄清。

---

## 六、Degrade / Rewrite / Stop 的差异

| 行为 | 含义 |
|------|------|
| **Degrade** | 降级执行（如：只做部分能力，明确告知用户） |
| **Rewrite** | 建议用户改写输入后重试 |
| **Stop** | 中止，不执行，返回拒绝话术 |

**选用原则**：由 Policy 显式定义，不交给 LLM 自判。
