## 一、Router 负责

| 职责 | 说明 |
|------|------|
| **接收 execution_id** | 唯一入口；不接收 task_id，不参与 Task 生命周期 |
| **Task Decomposition** | 基于 execution 的 payload 做任务分解；输出 capability_chain、memory_refs（与 FSD 04.2、架构 2.6 对齐） |
| **Capability 编排** | 按 Capability Registry 的 allowlist 调用 capability；构造 CapabilityInput Envelope，接收 CapabilityOutput |
| **审计写入** | execution_id、capability_id、capability_version、status、failure_mode、recommendations 等写入 audit |
| **Execution 状态更新** | executing → completed / failed；与 03.2 Execution 生命周期一致 |

---

## 二、Router 不负责

- **Task 生命周期**：Task 的创建、调度、取消由 Entry/Orchestration 负责。
- **业务逻辑判断**：业务正确性、Intent 深度理解在 Entry/Intent Analyzer；Router 只做确定性编排。
- **错误修复**：Capability 失败时写入 audit、可选回退策略，不直接修复上游数据。
- **Evolution 直接改写**：Evolution 对 Router 仅通过 Recommendation（见 FSD 04.5、Integration Spec 4.3）；禁止黑箱、必须审计。

---

## 三、与 Technical Architecture 衔接

- **Technical Architecture**：Intent Analyzer 输出 intent + slots，严禁输出 capability_chain；Router 接收 execution_id 后独立做 Task Decomposition。
- **Integration Spec**：Router 只接收 execution_id；CapabilityInput/CapabilityOutput Envelope、experience_recommendations 与 context 分离；Grey Area、Evolution 宪法级约束见 IS 1.3.4、4.3。
