## 1. Purpose

This matrix validates only `Step 2 - Session Harness`.

It does not validate:
- Step 3 capability matching
- final business answer correctness
- gate/router/capability execution

It validates only:
1. `session_resolution`
2. `turn_relation`
3. `conversation_context`
4. `registry_context`
5. `memory_runtime`
6. `policy_decision`
7. `effective_query`
8. `context_ready`

## 2. Pass rule

Step 2 V2 passes only if:
1. relation judgment is LLM-led on the hot path
2. TG and Web share the same taxonomy and acceptance semantics
3. old clarification contamination is blocked
4. `ContextPacket` shape is stable across all cases
5. heuristics, if used, are explicitly marked as fallback

## 3. Case template

Each case must record:
- `Test ID`
- `Channel`
- `Raw Query`
- `Preloaded Session State`
- `Expected session_resolution`
- `Expected turn_relation`
- `Expected policy_decision`
- `Expected context inheritance`
- `Expected recall behavior`
- `Expected prohibited behavior`
- `Failure attribution`

## 12. Immediate fail conditions

If any of the following occurs, Step 2 is an automatic FAIL:

1. new question becomes old clarification reply
2. correction becomes continuation
3. history callback becomes generic follow-up
4. Step 2 emits capability match or intent label
5. `ContextPacket` is incomplete or unstable
6. TG and Web produce different relation semantics for the same fixture
7. regex/keyword fallback silently replaces the LLM hot path without marking fallback

## 13. Minimum acceptance threshold

Step 2 V2 is only accepted if:
1. every group A-H has at least one passing positive case
2. clarification contamination cases are all green
3. cross-channel consistency cases are green
4. no automatic fail condition appears
5. the runtime output artifact remains exactly `ContextPacket`
