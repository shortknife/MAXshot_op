1. `gate_result=continue_chat` -> reject sealing
2. `out_of_scope` payload -> reject sealing
3. `gate_result=pass` -> allow sealing
4. `gate_result=require_confirmation` -> allow sealing with pending confirmation

1. read-only capability + `pass` -> `created`
2. side-effect capability + `require_confirmation` -> `pending_confirmation`
3. caller-supplied `require_confirmation=false` cannot override side-effect risk

1. > `MAX_MATCHED_CAPABILITIES` -> reject
2. missing primary capability when candidates exist -> reject or normalize deterministically
3. same `idempotency_key` -> returns prior task/execution pair
