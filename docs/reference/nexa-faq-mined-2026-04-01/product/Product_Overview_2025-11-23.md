## 1. 产品介绍

### 1.1 产品定位

**Nexa** 是一个 **B2B SaaS 智能 FAQ Bot 平台**，基于 X402 支付协议和 RAG 技术，为项目方提供智能 FAQ 服务基础设施，让项目方能够为终端用户提供高质量的 FAQ 问答服务。

**产品Slogan**：Connect Knowledge, Instant Answers（连接知识，即时回答）  
**品牌调性**：现代、智能、连接、未来

### 1.2 目标用户

**主要用户（B端客户）**：
- **项目方（企业用户）**：需要为终端用户提供 FAQ 服务的项目方
- **使用场景**：产品咨询、技术支持、售后支持
- **付费方式**：按问答次数付费（X402 协议）

**终端用户（C端用户）**：
- **项目方的用户**：使用项目方提供的 FAQ 服务
- **使用方式**：免费使用 FAQ 服务
- **接入方式**：网页端、Telegram Bot、Discord Bot、X（Twitter）被动回复

### 1.3 核心价值

**对项目方**：
1. **降低FAQ服务成本**：无需自建FAQ系统，按需付费
2. **提升服务质量**：基于RAG技术的智能问答，回答更准确、自然
3. **多端统一服务**：一次配置，多端部署（网页/TG/DC/X）
4. **区块链原生支付**：X402协议，微支付，降低使用门槛

**对终端用户**：
1. **免费使用**：无需付费即可获得FAQ服务
2. **多端接入**：通过熟悉的平台（网页/TG/DC/X）使用服务
3. **即时响应**：智能问答，快速获得答案

### 1.4 产品差异化

| 维度 | 传统FAQ Bot | 现代LLM/RAG FAQ Bot | **我们的产品** |
|------|------------|---------------------|--------------|
| **商业模式** | 订阅制（按月/年） | 订阅制（按月/年） | ✅ **X402按次付费** |
| **用户角色** | 单一角色（企业自用） | 单一角色（企业自用） | ✅ **双角色（B2B SaaS）** |
| **内容范围** | 可回答范围较宽泛 | 可回答范围较宽泛 | ✅ **严格限制在知识库范围内** |
| **边界控制** | 转人工或模糊回答 | 可能有幻觉问题 | ✅ **双重检测+明确拒绝** |
| **支付方式** | 传统支付（信用卡等） | 传统支付（信用卡等） | ✅ **区块链原生支付（X402）** |

**核心创新点**：
1. **严格的内容边界控制**：仅回答知识库内容，避免幻觉
2. **智能超出范围检测**：双重判断（向量+LLM），避免无效收费
3. **区块链原生支付**：X402协议，微支付，降低门槛
4. **B2B SaaS模式**：项目方付费，终端用户免费

### 2.2 功能清单

#### M1 知识库管理
- **知识库上传**：Web界面上传Markdown文档
- **自动处理**：文档自动切片、向量化、存储到Supabase向量库
- **质检流程**：四阶段LLM质检（Stage0/1/2/3/4）
- **知识库预览**：查看已上传的知识库内容和向量化状态

#### M2 FAQ引擎
- **RAG检索**：基于向量检索的智能问答
- **多端适配**：支持Web、Telegram、Discord、X（Twitter）
- **Fallback策略**：8种Fallback原因，确保用户体验
- **引用检测**：双重检测（向量+LLM），避免无效收费

#### M3 支付与计费
- **X402支付**：基于X402协议的微支付
- **按次扣费**：问答成功后按次扣费
- **余额校验**：扣费前余额校验，确保支付成功
- **对账流程**：每日对账，确保账本准确

#### M4 渠道扩展（Telegram）
- **Telegram广播**：FAQ答案在Telegram群组/频道广播
- **审批流程**：默认开启审批，确保内容质量
- **重试策略**：最多2次重试，pending处理
- **速率限制**：每日100次广播，超过90次提醒

#### M5 安全与监控
- **安全事件检测**：敏感内容命中、FAQ连续失败、权限/余额异常
- **节流策略**：同项目+同事件5分钟最多3条
- **告警流程**：用户拒答模板、运营邮件模板
- **SLA管理**：高优事件4h/24h、连续失败24h/72h

#### M6 Dashboard & Insights
- **FAQ指标卡**：回答数、成功率、引用覆盖率、Fallback占比
- **事件看板**：安全事件、支付异常、渠道待办
- **成本&SLA卡片**：LLM成本、扣费成功率、SLA超时次数
- **角色化提示**：PM、Ops、QA、Finance四角色提示语

### 3.3 数据流

#### 知识库上传流程
```
项目方上传Markdown
  → n8n Webhook接收
  → 文档解析（Frontmatter + 正文）
  → 文档切片（Chunk）
  → 向量化（Gemini Embeddings）
  → 质检（Stage0/1/2/3/4）
  → 入库（Supabase Vector Store）
  → Dashboard显示结果
```

#### FAQ问答流程
```
终端用户提问
  → n8n Webhook接收
  → 问题归一化
  → RAG检索（向量检索Top-K=3）
  → LLM生成（DeepSeek-V2）
  → 输出格式化（JSON Schema）
  → 渠道适配（Web/TG/DC/X）
  → 响应验证（格式/OOS/敏感）
  → 支付扣费（X402）
  → 日志记录（faq_response_logs）
  → 结果交付
```

#### 支付扣费流程
```
FAQ回答成功
  → 余额校验（X402 Facilitator）
  → 扣费请求（X402协议）
  → 交易记录（billing_transactions）
  → 账本更新（billing_ledger）
  → 日志记录（metrics_json.billing）
  → 结果反馈
```

## 7. 常见问题（Q&A）

### 6.1 产品相关问题

**Q1：产品的核心价值是什么？**
A：产品的核心价值是为项目方提供智能FAQ服务基础设施，降低FAQ服务成本，提升服务质量，支持多端统一服务，采用区块链原生支付。

**Q2：产品的目标用户是谁？**
A：主要用户是项目方（B端客户），终端用户（C端用户）免费使用FAQ服务。

**Q3：产品的商业模式是什么？**
A：采用X402协议按次付费，项目方付费，终端用户免费。

---

### 6.2 技术相关问题

**Q1：使用的技术栈是什么？**
A：数据库使用Supabase（PostgreSQL + pgvector），工作流使用n8n，向量模型使用Gemini 1536维，LLM模型使用DeepSeek-V2，支付协议使用X402，渠道使用Telegram MVP。

**Q2：系统架构是怎样的？**
A：系统采用三层架构：多端适配层、核心服务层（n8n工作流）、数据存储层（Supabase）。

**Q3：数据流是怎样的？**
A：数据流包括知识库上传流程、FAQ问答流程、支付扣费流程，所有流程通过n8n工作流编排。

---

### 6.3 构建相关问题

**Q1：如何构建产品？**
A：构建过程包括开发环境搭建、数据库初始化、工作流开发、测试验证、部署上线五个阶段。

**Q2：开发流程是怎样的？**
A：开发流程按照M1–M6模块顺序进行，每个模块包括Data、Workflow、AI、Prompt、QA五类FSD文档。

**Q3：测试流程是怎样的？**
A：测试流程包括单元测试、集成测试、端到端测试、性能测试、安全测试五个阶段。
