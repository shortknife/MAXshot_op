# MAXshot 团队记忆存储机制说明（四层架构）

> 与二黑团队 cognitive-memory 设计对齐，保证「记忆结构一致」。  
> 维护：LEE | 2026-03

---

## 四层记忆架构

```
┌─────────────────────────────────────────────────────────┐
│ 1. CORE MEMORY (核心记忆) - MEMORY.md                    │
│    - 长期身份、职责、原则、工作流程                       │
│    - 永远在 context 中 (~3K tokens)                       │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 2. EPISODIC MEMORY (情景记忆) - memory/<部门>/YYYY-MM-DD.md │
│    - 按部门：产品/开发/管理/OpenClaw；每日事件、进展、结论 │
│    - 按日期追加，可回溯查询                              │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 3. SEMANTIC MEMORY (语义记忆) - memory/ontology/         │
│    - 知识图谱、实体关系                                  │
│    - Person/Project/Technology 等结构化数据              │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 4. PROCEDURAL MEMORY (程序记忆) - memory/procedures/    │
│    - 学到的工作流程、技能                                │
│    - 如「情报交付流程」「通知确认机制」「迁移步骤」      │
└─────────────────────────────────────────────────────────┘
```

---

## 各层用途与写入约定

| 层 | 路径 | 存什么 | 谁写 | 何时写 |
|----|------|--------|------|--------|
| 核心 | `MEMORY.md` | 团队整体：边界、Rule、Agent 调整、关键文档索引 | **LEE 独维护** | 长期决策/规则落地时 |
| 情景 | `memory/产品/`、`memory/开发/`、`memory/管理/`、`memory/OpenClaw/` 下 YYYY-MM-DD.md | 当日进展、交付、阻塞、结论（按部门） | 各角色写本部门；LEE 写管理 | 当日或次日 |
| 语义 | `memory/ontology/` | 实体、关系、知识图谱（可选） | LEE / 按需 | 有结构化实体时 |
| 程序 | `memory/procedures/` | 固化的工作流程、SOP 摘要 | LEE / 角色 | 流程稳定后沉淀 |

## 分门别类速查表（内容→层）

| 内容类型 | 放哪一层 | 路径 / 示例 |
|----------|----------|--------------|
| 组织架构、长期决策、红线、关键文档表 | 核心 | `MEMORY.md` §一～四 |
| 某日的进展、交付、阻塞、会议结论 | 情景 | `memory/产品/`、`memory/开发/`、`memory/管理/`、`memory/OpenClaw/` 下当日 YYYY-MM-DD.md，按部门写入 |
| 实体与关系（人/项目/系统/技术栈） | 语义 | `memory/ontology/*.md`（按主题建文件） |
| 固化步骤（排查、迁移、交付、通知流程） | 程序 | `memory/procedures/*.md`（一个流程一文件） |
| 跨日可回溯的「谁在何时做了什么」 | 情景 | `memory/` 按日期，多日则多文件 |
| 需要长期引用的「我们决定了 X」 | 核心 | `MEMORY.md` §二 重要决策 |
| 需要按路径回忆的正式文档 | 核心 | `MEMORY.md` §四 或 `DOC_INDEX.md` |

**不确定时**：能按日期讲的 → 情景；能当「我们永远这样」的 → 核心；能当步骤复用的 → 程序；实体/关系表 → 语义。

## 记忆分工与边界（方案 A）

团队记忆**基于规则、基于团队内部协作**（如文档约定、谁写哪块）。情景记忆按**部门**分子目录，每人写本部门，团队整体由 LEE 维护。

### 四部门与谁写

| 部门 | 路径 | 谁写 | 写什么 |
|------|------|------|--------|
| **产品部** | `memory/产品/YYYY-MM-DD.md` | 产品侧（如 LEO / Decision_Layer） | 产品决策、批次结论、优先级、验收口径、FSD/产品文档相关结论 |
| **开发部** | `memory/开发/YYYY-MM-DD.md` | 开发侧（Alex / Sam / Lily 等） | 技术实现、交付物、阻塞、验收结果、技术决策 |
| **管理部** | `memory/管理/YYYY-MM-DD.md` | 你 + LEE | 你我的聊天里需留痕的规则/流程/决策；LEE 对团队整体的维护 |
| **OpenClaw 部** | `memory/OpenClaw/YYYY-MM-DD.md` | 负责二黑/大乖的角色 | 二黑、大乖、OpenClaw 的辅助决策、配置共识、与主产品线的对接结论 |

### 团队整体记忆（仅 LEE）

- **MEMORY.md** 只由 **LEE** 填写与更新，作为团队整体记忆：边界、Rule 更新、Agent 调整、协作与文档约定、红线。
- 各角色不直接改 MEMORY.md；若有需纳入团队整体的决策或规则，由 LEE 提炼后写入。

### 边界原则

- **产品部** ≠ 查 LEO 私人记忆，只存产品线共识/决策。
- **管理部** = 规则与协作 + 你我的管理/运营类沟通；团队整体（边界、Rule、Agent）仅在 MEMORY.md，由 LEE 维护。
- 只有**需要团队共享**的内容才进入团队记忆；其余留在对话或各角色 4.Working 即可。

## 每日整理、每周记忆处理与 LEE 兜底

- **每日整理记忆**：各角色在当日或次日把本部门当日进展、交付、阻塞、结论写入 `memory/<本部门>/YYYY-MM-DD.md`，避免堆积。
- **每周记忆处理**：**LEE** 每周做一次整体记忆处理，将零散或错位内容**规范到各自位置**（该进产品部的归产品、该进管理的归管理、该进 MEMORY.md 的提炼进核心），保证不乱、可回溯。历史曾写在 `memory/` 根下 YYYY-MM-DD.md 的日刊，可在每周整理时按内容迁移到对应部门目录。
- **LEE 负责兜底记忆**：记忆最终归位、结构清晰、可被唤醒时正确读取，由 LEE 兜底；发现漏记、错放或需合并时，由 LEE 补写或迁移。

## 聊天与协作文档约定

- **全量聊天记录**：不要求对历史聊天做全量导出与记忆整理；重要结论靠日常「记一下」与对话小结写入 memory，老旧聊天可随时删除。
- **5.Collaboration**：多为过程文档，只对**有重点信息且尚未进 memory/MEMORY** 的做提炼后归档；其余可只归档不提炼。
- **重新唤醒小伙伴前**：做一次 memory 整理（查 memory/ 近期 + MEMORY 重要决策），即可把当前共识与进展带入新对话，无需依赖旧聊天记录。

## 小伙伴如何用记忆系统（优先记忆、少搜文档）

**原则**：回复/分工/验收**先依据记忆**，再按需打开 DOC_INDEX 里的正式文档；**不要**用全量搜索替代记忆。

| 场景 | 怎么做 | 不怎么做 |
|------|--------|----------|
| **唤醒 / 开工** | 先读 `MEMORY.md` + `memory/产品/`、`memory/开发/`、`memory/管理/`、`memory/OpenClaw/` 下今日与昨日 YYYY-MM-DD.md；需要时扫 `DOC_INDEX.md` | 不要一上来就全库搜 md/json |
| **被问决策 / 进展 / 谁负责** | 从 MEMORY「重要决策」、各部门 memory 日刊里答；没有再查具体文档 | 不要先搜 5.Collaboration、4.Working 再拼答案 |
| **要找某份正式文档** | 用 `DOC_INDEX.md` 查路径（如 FSD 入口、产品架构、当前批次跟踪），按路径打开 | 不要用关键词全量搜整个仓库 |
| **写结论 / 记一下** | 讨论有结论、交付、阻塞时写入 `memory/当日` 或 MEMORY；路径见写入指南 | 不要只留在聊天或零散 md 里 |

**一句话**：记忆是「我们决定了什么、做到了什么、谁管什么」的单一出口；DOC_INDEX 是「要查的正式文档在哪儿」的入口；只有记忆和 DOC_INDEX 指向的文档才值得打开，其余过程文档在 1.Archive，不必搜。

## Cursor 侧：任务需要时自动加载记忆（类 Skill）

在 Cursor 中，`.cursor/rules/team-memory.mdc` 规定：**当任务需要团队/角色上下文时**（在角色工作区内、团队相关任务、未提供明确背景、或回复需依据共识），Agent 必须先静默读取 MEMORY.md + memory/产品/、memory/开发/、memory/管理/、memory/OpenClaw/ 下今日+昨日 YYYY-MM-DD.md，再依据记忆执行。这样「和 Alex/LEO/… 对话」时，只要落在触发条件内，就会先带历史记忆再回复，记忆体系可见效。触发条件与必读路径见该 rule 文件。

**口播约定**：回答团队相关/共识类问题时，若用到了记忆，须简短带一句「已据 MEMORY + memory/ 近期与下列文档作答」（或「已据团队记忆与下列文档作答」），便于验证「这次有走记忆」。见 team-memory.mdc § 执行方式。
