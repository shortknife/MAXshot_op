# Team_Memory 团队记忆

> **定位**：Cursor 开发环境下团队的持久化记忆，类似 OpenClaw 的 memory 机制。  
> **目的**：新对话自动获得团队上下文，无需每次手动总结历史对话。

---

## 四层记忆架构（与二黑一致）

| 层 | 路径 | 说明 |
|----|------|------|
| 1. 核心 | `MEMORY.md` | 长期共识、重要决策、红线、关键文档索引 |
| 2. 情景 | `memory/YYYY-MM-DD.md` | 每日进展、事件、结论摘要 |
| 3. 语义 | `memory/ontology/` | 知识图谱、实体关系（按需） |
| 4. 程序 | `memory/procedures/` | 固化工作流程、SOP 摘要（按需） |

详见 **`记忆存储机制说明_四层架构_2026-03.md`**。

## 使用方式

### 新对话时（AI/Agent）

1. **必读**：`MEMORY.md`（长期共识）
2. **必读**：`memory/` 下今日 + 昨日文件（近期进展）
3. 如需更早背景，再查阅具体日期或检索关键词

### 写入规则

| 内容类型 | 写入位置 | 示例 |
|----------|----------|------|
| 重要决策、长期共识 | `MEMORY.md` | 架构变更、SOP、红线 |
| 当日讨论、进展、待办 | `memory/YYYY-MM-DD.md` | 会议结论、开发进展、阻塞 |
| 个人工作日志 | 各角色 `4.Working/` | 不强制写入 Team_Memory |

### 维护责任

- **MEMORY.md**：LEE 主维护；各角色在重要决策后补充「重要决策」小节
- **memory/YYYY-MM-DD.md**：当日有讨论/进展的角色创建或补充；可多人共同维护

### 如何主动写入？

详见 **`Team_Memory_写入指南_小伙伴主动记_2026-02.md`**：触发点、低摩擦写法、AI 协助、习惯绑定。

### 怎么监控「有没有人写」？

详见 **`Memory_监控与检查_2026-02.md`**：你能做的 3 件事、和 Cursor 的配合方式（你说「检查最近几天 Memory 有没有漏记」我会跑一遍并给建议补记）。

### 关窗口 / 重启前：旧对话收口（先归档、默认不删）

- **SOP**：`旧对话收口_SOP_先归档不删_按月复盘_2026-03.md`（结论进 memory → 更新 Restart Pack → 过程稿移 `1.Archive/`，**暂不删除**；**每月**由 LEE 做一次归档复盘）。
- **Cursor Skill（v1.1+ 默认全自动）**：`.cursor/skills/team-memory-dialog-closeout/SKILL.md` — Boss **一句话**见 **`一键收口_给Boss用_2026-03.md`**；Agent 按步骤执行、**本地 git commit**（不 push）、**不逐项问你**，文末 **OK / NOT OK** 检测表（含 Git 项）。

## 与现有机制的关系

| 原机制 | 处理方式 |
|--------|----------|
| 各角色 `Conversation_History_Summary` | 逐步废止，由 Team_Memory 替代 |
| Handover_Report | 保留；关键结论可同步写入 MEMORY.md |
| AGENTS.md、PERSONA、TODO | 保留；Team_Memory 作**补充**，不替代角色定义 |

---

## 不放入共享目录的原因

- Team_Memory 是 **内部协作记忆**，不对外、不对 OpenClaw/二黑等暴露
- 放在 `00.System/` 下，与组织架构、系统级文档同层，便于统一管理
