## 二、逐项决议正文（供审视开发时引用）

### 决议 1：Side Effect 必须确认（pending_confirmation）

- **结论**：**Phase 1 仅 Admin OS 确认**；TG/Notion 内确认列为后续迭代；契约与设计留 extension 位。
- **约束**：
  - 发布类（Publisher）等 Side Effect：先写 task_executions 且 status=pending_confirmation，**用户仅在 Admin OS 确认**后才触发 Router/执行发布。
  - 在契约/设计文档中写明「Phase 1 确认入口 = Admin OS；TG/Notion 内确认 = 后续迭代」，并预留确认入口与超时策略的扩展。
- **审视要点**：开票或实现「发推/发布」时，必须体现「先 pending_confirmation → Admin OS 确认 → 再执行」；不得实现「TG 直接确认即发」除非明确列为后续迭代。

---

### 决议 2：Gate 双重角色

- **结论**：**显式拆成两个 Gate**——**Entry Gate** 与 **Capability Gate**（或 Contract Gate）。
- **约束**：
  - **Entry Gate**：放行/拒绝进系统、价值信息判断（无信息量可不进 capability）、entry_reject/soul_reject。
  - **Capability Gate**：G1–G6、结果验收、**contract_passed=false 一律不返回 success**；归属各 Capability 调用链内（如 Data Query Contract v2）。
  - 文档与实现均使用两个明确名词，避免混用「Gate 没过」指代不明。
- **审视要点**：设计/工票若写「Gate」，须标明是 Entry Gate 还是 Capability Gate；价值信息判断归属 Entry Gate，规则由产品/策略定。

---

### 决议 3：三入口分工（形态为第一维度）

- **结论**：**以输入形态为第一维度**；渠道名仅用于审计与 Output 适配。
- **约束**：
  - Entry 先定 **entry_type = raw_query | structured | timeline**，再按此分支：是否走 Intent、是否封印、防重锁。
  - **raw_query**（TG/Chat 文本）→ 必走 Intent；**structured**（定时/Notion/按钮）→ 跳过 Intent 直接封印；**timeline**（publishing_timeline 到期）→ 跳过 Intent、封印+防重锁。
  - TG/Notion/Admin OS 作为 **entry_channel** 用于审计与按 channel 分发；路由键 = entry_type。
- **审视要点**：实现 Entry/总编排时，必须有 entry_type 分支逻辑；文档中建议保留「渠道 → 默认形态」对照表（如 TG 文本→raw_query，Notion webhook→structured，timeline 扫描→timeline）。

---

### 决议 4：Output 以出口类型为一等公民

- **结论**：**Output 层以 01.3 出口类型为一等公民**；按 channel 仅为同一 exit_type 下的展示差异。
- **约束**：
  - Output 输入包含 **exit_type**（completed / failed / rejected / expired / user_abandoned / unknown / entry_reject / soul_reject）。
  - 每种 exit_type 有对应话术/行为（可先做最小集：completed、failed、entry_reject、rejected）；所有出口（含 Reject）须可追溯、符合 01.3。
- **审视要点**：设计「统一 Output」或开 Output 相关工票时，须包含 exit_type 与拒绝路径话术；不得仅做「成功结果按 channel 格式化」。

---

### 决议 5：Intent 严禁输出 capability_chain

- **结论**：**契约 + 运行时校验**；Intent 输出 schema 仅含 FSD 允许字段（如 intent_name、slots），禁止 capability_chain/capability_id；Router 前或 Intent 出口校验，违规则拒绝并打审计。
- **约束**：
  - 06.1：任务分解权 100% 在 Router；LLM 只做语义解析，输出仅含 intent + extracted_slots。
  - 实现：维护 Intent 输出 schema，在编排/代码中校验并丢弃或拒绝任何 capability_* 字段。
- **审视要点**：Intent 上 n8n 或代码时，须有输出 schema 文档与校验逻辑；Code Review 检查是否存在「Intent 直接输出 capability 相关字段并被 Router 使用」。

---

### 决议 6：业务数据优先（不混淆治理遥测）

- **结论**：**写进主流程与验收**；主产品查询答案仅来自 Business Data Plane（vault/execution/market/allocation）；Governance 仅控制面；验收含「不得用治理表当业务答案」的检查（含负面用例）。
- **约束**：
  - 主流程/架构图显式写：「主产品查询答案来自 Business Data Plane；Governance 不参与主答案。」
  - 验收用例：业务查询数据源白名单 + 负面用例（若答出 task_executions/audit 内容则失败）。
- **审视要点**：开票「业务查询」「Data Query」「主产品答案」时，须对齐 Business 表清单；不得静默使用 task_executions/audit_logs 作为用户可见答案来源。

---

### 决议 7：Execution 不可变、留痕、Evolution、Capability 不直连 DB（直接采纳）

- **Execution 不可变 + 留痕**：主流程或核心原则引用 00.2——Execution 启动时 Snapshot 冻结；所有决策/拒绝/确认写审计；不留痕 = Bug。
- **Evolution**：主流程或架构图加一句——Evolution 仅通过 Recommendation 影响 Router，不自动改逻辑。
- **Capability 不直连 DB**：Capability 开发标准/CC 总则写——数据仅通过 Router/context（含 memory_refs）获取，不直连 DB。

---

### 决议 8：对话 Session 与 Channel（小微组织适用）

- **结论**：**显式承认「会话」边界**，留痕可串起「同一轮对话」；**同 session 内是否严格 FIFO 排队**由产品策略决定，不做默认承诺。
- **约束**：
  - **Session 边界**：定义并落库「同一用户、同一通道、一定时间窗口内的请求」属同一 session（或由入口/前端传 session_id）；审计与 execution 关联 session_id（或等价），便于「这一串请求是一轮对话」可追溯。
  - **Channel**：保持三入口（TG / Admin OS / Notion）+ **entry_type** 为路由第一维度；新增通道 = 新增 Adapter + 约定 entry_type，不追求多通道数量。
  - **同 session 内顺序**：若产品要求「同一用户连续多句严格按序执行、不插队」，则实现 Per-Session FIFO（或同 session 串行化）；否则可允许同 session 并发，以 execution_id 各算各的，产品接受可能乱序。**由产品策略显式选择**，不在 FSD 默认写死。
  - **长对话/记忆**：主产品为「查业务 + 发布」，非开放域长闲聊；memory_refs + session_context 支撑指代与短链多轮即可；压缩刷新、长程聊天记忆列为可选增强，有明确需求再上。
- **审视要点**：Entry/留痕设计须能体现「谁、哪次对话」；新入口或新通道时检查 session_id（或等价）是否传递与落库；若出现「用户连续发多条顺序错乱」的投诉，再评估是否启用同 session FIFO。
