## 一、今天能帮你做什么（3 件具体的事）

### 1. 用自然语言查 DeFi 运营数据（主产品 · P0）

**你可以问**：
| 你说的话 | 系统回复（示例） |
|---------|---------------|
| 「现在有几个金库？分别部署在哪几个链？」 | 「目前有 3 个金库在运行，分别部署在 Arbitrum。」 |
| 「USDC 金库的 APY 是多少？」 | 「USDC 金库当前 24h 平均 APY 为 8.7%，数据来自链上快照。」 |
| 「最近 7 天 TVL 的变化趋势？」 | 「过去 7 天 TVL 从 $1.2M 增长至 $1.5M，增幅 25%。」 |
| 「哪个金库收益最高？」 | 「当前 APY 最高的金库是 Morpho-ETH-USDC，24h 平均 12.3%。」 |
| 「这个 Vault 最近一次 Rebalance 是什么时候？」 | 「最近一次 Rebalance 发生在 2026-02-25 03:12 UTC。」 |

**技术真相**：系统用 SQL 查你的真实业务数据（链上快照、金库记录、分配记录），LLM 只负责把 SQL 结果翻译成人话，不会编造数字。

**对应 Capability**：`capability.data_fact_query`（见 CC-02）  
**对应 FSD**：v5.1 §3 Ops 数据可信化、02.5 §二 #2

### 2. 生成有灵魂的社媒内容（副产品 · P1）

**你可以说**：
| 你说的话 | 系统回复（示例） |
|---------|---------------|
| 「帮我写一条关于 DeFi 收益策略的推文」 | 「[草稿] Yield isn't luck—it's discipline. Our vaults run 24/7... 标签: educational / defi / twitter_main」 |
| 「用教育风格写一条 Thread，讲金库运作原理」 | 「[Thread 草稿] 1/5 你知道 DeFi 金库是怎么帮你赚钱的吗？...」 |
| 「这周的内容复盘报告呢？」 | 「本周共发布 7 条内容，互动率最高的是周五 meme 风格推文（互动率 4.2%）...」 |

**注意**：生成的是**草稿**，不是直接发布。发布需要你在 Admin OS 点确认（Side Effect 确认机制）。系统会为每条内容打上四维标签（风格 / 时间 / 话题 / 渠道），用于后续归因分析。

**对应 Capability**：`capability.content_generator`（见 CC-04）+ `capability.publisher`（见 CC-03）  
**对应 FSD**：v5.1 §4 Marketing 策略闭环化、02.5 §二 #5/#6

### 3. 查产品文档和 FAQ（辅助 · P1）

**你可以问**：
| 你说的话 | 系统回复（示例） |
|---------|---------------|
| 「APY 是怎么计算的？」 | 「APY 基于 24h 链上收益年化计算，公式为 ... 来源: FAQ #12」 |
| 「MAXshot 的产品逻辑是什么？」 | 「MAXshot 是小微团队增长操作系统，核心是 Ops 数据可信化 + ... 来源: 产品文档 §1.1」 |

**对应 Capability**：`capability.product_doc_qna`（见 CC-05）  
**对应 FSD**：02.5 §二 #3

### 系统的边界：今天做不了什么

| 你的问题 | 系统行为 |
|---------|---------|
| 「帮我把 10 ETH 转到另一个金库」 | **拒绝**——涉及资金变更，超出系统权限（Soul 红线）|
| 「预测一下明天 BTC 价格」 | **拒绝**——不做预测，只查事实数据 |
| 「你好 / 今天天气怎么样？」 | **拒绝**——超出业务范围，友好告知 |
| 「详细配置一个新的 Campaign 策略」 | **引导到 Admin OS**——复杂配置不在 TG Bot 承载范围 |

## 二、你发一句话，到看到回复，中间发生了什么

以 Telegram 输入「现在有几个金库？分别部署在哪几个链？」为例：

```
你在 Telegram 发了这句话
     │
     ▼
① [入口接收] Telegram Bot 收到消息，标记为"自然语言输入"
     │
     ▼
② [理解你的意图] 系统分析这句话，判断你想问的是"运营数据查询"
   识别出：意图=查金库信息，需要的数据=金库数量+链分布
     │
     ▼
③ [安全检查] 确认这个问题在允许范围内（不涉及资金操作、不违反安全边界）
     │
     ▼
④ [查数据] 系统从业务数据库中查询：
   - 查金库表，统计活跃金库数量
   - 查分配记录，提取部署链信息
     │
     ▼
⑤ [组织回答] 把查到的数据翻译成人话：
   "目前有 3 个金库在运行，分别部署在 Arbitrum。"
     │
     ▼
⑥ [记录审计] 整个过程被记录（谁问的、什么时间、查了什么、返回了什么）
     │
     ▼
你在 Telegram 看到了回复
```

**关键设计原则**（所有角色需理解）：
- 数据来自真实数据库，不是 AI 编的（**真实**）
- 整个过程有完整记录，可以回放和审查（**可审计**）
- 如果数据库查不到答案，会告诉你"无法回答"，不会瞎猜（**不猜不编**）

**对应 FSD 的工程视角**（开发人员可展开）：

| 用户感知步骤 | 工程实现 | FSD 文档 |
|------------|---------|---------|
| ① 入口接收 | Entry Adapter（entry_type=raw_query） | 01.1、08.4 §一 |
| ② 理解意图 | Intent Analyzer（LLM 解析 → intent + slots） | 06.1、06.4 |
| ③ 安全检查 | Entry Gate + Soul Filter | 01.2、05.1/05.2 |
| ④ 查数据 | Router → DataFactQuery Capability（SQL Engine 三级策略） | 02.5、CC-02、v5.1 §3.4 |
| ⑤ 组织回答 | Answer Formatter + M7 回传 | CC-02 §6、08.4 |
| ⑥ 记录审计 | audit_logs + task_executions | 09.1、09.2 |

## 三、三个入口，各做各的事

| 入口 | 谁用 | 干什么 | 典型场景 |
|------|------|--------|---------|
| **Telegram Bot** | 运营 / 创始人 | 用自然语言快速查数据、下简单指令 | 「金库 APY 多少？」「帮我写条推文」 |
| **Admin OS** | 运营 / 管理员 | 看审计日志、确认发布、管理 Capability、调试 | 确认发布推文、查看某次执行的完整路径 |
| **Notion** | 内容团队 | 管理内容 pipeline、排期、批量任务 | 维护内容日历、设置定时发布 |

**边界明确**：
- 在 Telegram 问「帮我配置一个 Campaign」→ 系统会说「请在 Admin OS 中操作」
- 在 Admin OS 想对话「金库 APY 多少？」→ 应该去 Telegram 问

## 五、半年后会是什么样

### v5.1（当前执行版）→ v5.2（下一版方向）

| 维度 | 今天（v5.1） | 半年后（v5.2 方向） |
|------|-------------|------------------|
| **理解能力** | 正则模板 + LLM 兜底 | LLM 为主理解引擎，零代码扩展新问法 |
| **回答质量** | 数据 + 格式化 | 数据 + 解释 + 洞察 + 建议 |
| **记忆** | 单次对话内 | 跨对话记忆——记住你关注的金库、常问的问题、偏好 |
| **主动性** | 等你问 | 主动推送异常（APY 波动 > 30%）、策略建议、日报 |
| **人格** | 查询工具 | 智能管家——有温度、有洞察、简洁但不冷漠 |

### 你今天做的事，未来怎么体现

| 今天做的 | 半年后的体现 |
|---------|------------|
| 每天在 TG 查金库数据 | 系统学会了你的关注点，主动推送日报 |
| 用 ContentGenerator 生成推文 | 归因分析告诉你「周五 meme 风格效果最好」 |
| 在 schema_index 中添加新视图 | LLM 自动理解新数据，无需写模板 |
| 发布内容并记录标签 | 周期复盘报告基于这些标签给出策略建议 |

**对应 FSD**：v5.2 产品方向备忘录 §一 ～ §六

## 六、如何扩展：新增一个 Capability 需要什么

> 场景：我们要新增「推特互动管理」能力（自动回复 mention、管理 bookmarks）

### 6 步接入 SOP

| 步骤 | 做什么 | 谁做 | 交付物 |
|------|--------|------|--------|
| **1. 产品定义** | 写 Capability Contract（CC-xx）：一句话定义、输入输出、验收标准、场景对话示例 | LEO | `CC_07_TwitterInteraction.md` |
| **2. Intent 注册** | 在 Intent Analyzer 的合法 intent 列表中加入新 intent 名称 + 描述 + 触发示例 | John + Alex | 06.4 `intent_schema` 更新 |
| **3. Registry 注册** | 在 Capability Registry 代码中注册新 handler | Sam | `admin-os/lib/capabilities/index.ts` |
| **4. 实现** | 按 CC 契约实现 Capability 逻辑 | Sam / Alex | Capability 代码 / n8n 工作流 |
| **5. 测试验收** | 用 CC 中的场景对话示例做端到端测试 | Lily | 测试报告 |
| **6. 更新本文档** | 在 §四 Capability 目录中加一行 | LEO | 本文档更新 |

**必须同步**的绑定关系（不做就会断裂）：
- Capability Registry 加了 → Intent schema **必须**同步加（否则用户问了，Intent 不认识）
- Intent schema 加了 → 06.4 Prompt 的合法 intent 列表**必须**同步加（否则 LLM 不会输出这个 intent）
- CC 契约写了 → 本文档 §四 目录**必须**同步加（否则 PM / 用户看不到这个能力）

**对应 FSD**：02.5 §三 Registry 门槛、06.1 §2.3 NLU

## 七、隐式耦合清单（跨模块绑定关系）

> 以下是「A 改了必须同步改 B」的清单。如果只改了 A 没改 B，系统会断裂。

| # | A（上游变更） | B（必须同步变更） | 断裂后果 | 对应 FSD |
|---|-------------|-----------------|---------|---------|
| 1 | Capability Registry 加新能力 | Intent schema 加对应 intent | 用户问了，系统不知道该路由到哪 | 02.5 ↔ 06.1/06.4 |
| 2 | Intent schema 加新 intent | `memory_layer_context` 加描述 | LLM 不知道有这个合法 intent，可能输出 unknown | 06.4 §2.4 |
| 3 | 新增视图/表到 schema_index | Tier1 模板 + Tier2 白名单同步 | SQL 查不了新表，或查了但被白名单拦截 | CC-02 §2.1 |
| 4 | ContentGenerator 输出标签变更 | Attribution / Cycle Report 同步 | 标签打了但归因分析不认识新标签 | CC-04 §3 ↔ v5.1 §4.2 |
| 5 | 新增 Side Effect 类 Capability | Admin OS 确认界面同步 | 发布请求进了 pending_confirmation 但没有确认 UI | CC-03 §5 |
| 6 | 新增 entry_channel（如 Discord） | Entry Adapter + Output 适配同步 | 新渠道消息进不来，或回复格式错 | 01.1 ↔ 08.4 |
| 7 | Soul 红线变更 | Intent Analyzer + Entry Gate 同步 | 本该拒绝的问题放行了 | 05.2 ↔ 06.1 |
