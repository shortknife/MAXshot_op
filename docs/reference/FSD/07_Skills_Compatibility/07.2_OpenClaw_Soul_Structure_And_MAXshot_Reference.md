## 一、OpenClaw 灵魂三件套（来源：模板与说明）

| 文件 | 作用 | 类比 | 模板路径 |
|------|------|------|----------|
| **SOUL.md** | 定义助手的性格、边界、Vibe | 基因 + 教养 | `openclaw/docs/reference/templates/SOUL.md` |
| **USER.md** | 描述「你在帮谁」——名字、称呼、时区、项目、偏好 | 简历 + 日记 | `openclaw/docs/reference/templates/USER.md` |
| **AGENTS.md** | 设定工作习惯、边界、每次启动必读顺序 | 员工手册 | `openclaw/docs/reference/templates/AGENTS.md` |

## 二、三文件如何形成（Onboarding 流程）

**首次运行**（`onboarding.md`）：

1. **Seed**：写入 `AGENTS.md`、`BOOTSTRAP.md`、`IDENTITY.md`、`USER.md`
2. **Q&A 仪式**：与用户对话，逐步填满
3. **产出**：写入 `IDENTITY.md`、`USER.md`、**SOUL.md**（一起讨论「什么重要、希望你如何表现、边界」）
4. **收尾**：删除 `BOOTSTRAP.md`，之后不再运行

**AGENTS.md 中的强制顺序**（Every Session）：

```
1. Read SOUL.md — this is who you are
2. Read USER.md — this is who you're helping
3. Read memory/YYYY-MM-DD.md (today + yesterday)
4. (主会话) Read MEMORY.md
```

→ **SOUL 先于一切**；USER 定义服务对象；AGENTS 定义「怎么干活」。

## 三、各文件核心内容摘要

### SOUL.md（Who You Are）

- **Core Truths**：真诚助人、有观点、主动尝试、用能力建立信任、记住是「客人」
- **Boundaries**：隐私不外泄、外部动作前先问、不发半成品到消息渠道
- **Vibe**：「Be the assistant you'd actually want to talk to」
- **Continuity**：每次会话读 SOUL；可进化，改了要告诉用户

### USER.md（About Your Human）

- Name, What to call them, Pronouns, Timezone, Notes
- **Context**：他们在乎什么、在做什么项目、什么让他们烦/开心（随对话积累）

### AGENTS.md（Your Workspace）

- **First Run**：若有 BOOTSTRAP，先走出生仪式
- **Every Session**：SOUL → USER → memory，固定顺序
- **Memory**：daily / MEMORY 的读写规则
- **Safety**：不泄露隐私、不随便执行破坏性命令
- **External vs Internal**：哪些可自由做、哪些要先问
- **Group Chats**：何时说话、何时沉默、反应方式
- **Tools / Heartbeats**：技能调用、主动巡检规则

## 四、MAXshot 如何借鉴（产品视角）

**原则**：MAXshot 不是「个人助理对话平台」，但**主文件的三层结构**（身份 / 对象 / 工作方式）可以沿用。

### 4.1 映射建议

| OpenClaw | MAXshot 等价物 | 说明 |
|----------|----------------|------|
| **SOUL.md** | **MAXshot_SOUL.md**（或架构中的 Soul 小节） | MAXshot 是谁、语气、边界；偏品牌人格 |
| **USER.md** | **服务对象定义**（requester_role、channel、运营/运维角色） | 谁在用 MAXshot、在什么场景用 |
| **AGENTS.md** | **各角色 AGENTS.md + 架构 Guardrails** | 工作习惯、禁止项、必读顺序 |

### 4.2 关键借鉴点

1. **加载顺序**  
   对人输出的 LLM（如 Intent 澄清、确认话术）启动时：  
   - 先读 Soul 摘要（MAXshot 是谁、边界）  
   - 再读「当前服务对象」摘要（来自 requester/channel）  
   - 再读 memory 与规则

2. **形成方式**  
   - OpenClaw：通过 BOOTSTRAP 对话生成 SOUL/USER  
   - MAXshot：产品/运营**显式编写**，无需对话生成；可作为 Prompt 注入或独立文档，定期评审更新

3. **AGENTS 的角色**  
   - OpenClaw：单 Agent 的「员工手册」  
   - MAXshot：多角色（LEO、Cat、Alex…）各有 AGENTS.md；**架构 Guardrails** 可视为「全系统 AGENTS」——Router 红线、禁止项等

### 4.3 落地最小集

| 动作 | 产出 |
|------|------|
| 起草 **MAXshot_SOUL** 初版 | 1 页：定位、Core Truths、Boundaries、Vibe |
| 定义 **MAXshot_USER**（服务对象） | requester_role、channel、典型使用场景的简要画像 |
| 在 Intent Analyzer / 澄清 Prompt 中注入 | Soul 摘要 + 服务对象摘要，作为 System 或前缀 |
