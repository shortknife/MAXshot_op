# 03.3 不可变快照模型 ⭐

> **用途**：防止工程师「偷懒 join 最新表」

---

## 一、核心原则

**Task 是可变配置，Execution 是不可变历史。**

**Snapshot = Execution 启动时冻结的一切上下文。**

---

## 二、含义

- Router 接收 execution_id 后，从 `task_executions` 读取的上下文，即该 Execution 的 **Snapshot**
- 后续 Task 表、Memory 表、配置表如有更新，**不影响已启动的 Execution**
- 禁止在 Execution 执行过程中，通过 join 最新表来「刷新」上下文

---

## 三、实现约束

- Execution 启动时，将所需 context（如 memory_refs、task 元数据）**复制**到执行上下文
- 执行期间不再次查询可能变化的表（除非显式设计为「实时拉取」且文档化）

---

## 四、产品决议（本版唯一执行）

**决议 7（Execution 不可变、留痕、Evolution、Capability 不直连 DB）**  
- **Execution 不可变 + 留痕**：主流程引用 00.2——Execution 启动时 Snapshot 冻结；所有决策/拒绝/确认写审计；不留痕 = Bug。  
- **Evolution**：仅通过 Recommendation 影响 Router，不自动改逻辑。  
- **Capability 不直连 DB**：数据仅通过 Router/context 获取（见 02.1 §四、§五）。  
- **开发/验收要点**：实现不得在 Execution 执行中 join 最新表改写历史；审计必写；Evolution 改动 Router 仅通过可审计的 Recommendation。
