## 1. 设计目的

EntryRequest 是从「外部入口」到 MAXshot Orchestration 的统一请求结构，是 **Intent Analyzer / Sealer / Router** 的共同语言。定义清晰、稳定的 EntryRequest Schema，是后续所有 Capability 与 Router 设计的前提。

本 Schema 只关心「一条进入系统的业务请求」在 **Orchestration 视角** 的形态，不绑定具体传输协议（HTTP、Webhook、n8n 节点等）。

---

## 2. 顶层结构 (Top-level Shape)

```json
{
  "entry_id": "string (UUID)",
  "entry_type": "raw_query | structured | timeline",
  "requester_id": "string | null",
  "entry_channel": "telegram | admin_os | notion | system | other",
  "idempotency_key": "string | null",
  "payload": {},
  "metadata": {},
  "created_at": "ISO8601 timestamp"
}
```

### 2.1 字段说明

- **entry_id**: 系统内部为 EntryRequest 生成的唯一 ID，通常与 `task_id` 或 `execution_id` 关联。
- **entry_type**:
  - `raw_query`: 需要 Intent Analyzer 进行自然语言解析；
  - `structured`: 已由上游生成结构化参数，可跳过 Intent Analyzer；
  - `timeline`: 由定时扫描触发（如 publishing_timeline），直接进入封印与防重逻辑。
- **requester_id**: 业务侧认为的「发起人」标识，如用户 ID、Bot ID 等；若未知可为 null。
- **entry_channel**: 入口通道，如 Telegram、Admin OS、Notion、System Job 等。
- **idempotency_key**: 幂等键，供 Sealer / Router 在写库时做去重；可为空。
- **payload**: 业务负载主体，不同 entry_type 的语义不同（见 §3）。
- **metadata**: 附加信息，如 ip、user_agent、chat_id、raw_update 等，不进入业务决策核心，但可用于审计与排错。
- **created_at**: 进入 Orchestration 层的时间戳，由 Sealer 统一生成。

## 3. 各 entry_type 下 payload 的形态

### 3.1 raw_query

```json
{
  "payload": {
    "raw_query": "string",
    "language": "string | null",
    "session_context": {},
    "intent_ir": null
  }
}
```

- **raw_query**: 原始用户文本，经过基础清洗（去掉控制字符、前后空格）。
- **language**: 可选语言标记，如 `en`, `zh-CN`，方便后续多语言支持。
- **session_context**: 会话快照，由 Context Load 生成，供 Intent Analyzer 使用；形态见 06.3。
- **intent_ir**: Intent Intermediate Representation，初始为 null，由 Intent Analyzer 写入（intent + extracted_slots）。

### 3.2 structured

```json
{
  "payload": {
    "intent": "string",
    "slots": {},
    "source": "scheduler | admin_action | sync_job",
    "extra": {}
  }
}
```

- **intent**: 上游已确定的意图名，如 `ops_query`, `publish_tweet`。
- **slots**: 上游已经填好的槽位，如时间区间、对象 ID 等。
- **source**: 结构化请求来源类型（定时任务、后台操作等）。
- **extra**: 额外信息，由具体 Capability 决定是否使用。

### 3.3 timeline

```json
{
  "payload": {
    "timeline_id": "string",
    "scheduled_at": "ISO8601 timestamp",
    "intent": "string",
    "slots": {},
    "retry_count": 0
  }
}
```

- **timeline_id**: 对应 timeline 表主键，用于防重与重试。
- **scheduled_at**: 计划执行时间。
- **intent / slots**: 与 structured 相同，表示已经确定的行动。
- **retry_count**: 当前重试次数，供 Router/Capability 做幂等或降级处理。

## 4. EntryRequest 与下游表结构的映射

### 4.1 与 tasks / task_executions 的关系（建议）

- `tasks.entry_type` ← EntryRequest.entry_type
- `tasks.entry_channel` ← EntryRequest.entry_channel
- `tasks.requester_id` ← EntryRequest.requester_id
- `task_executions.payload` ← EntryRequest.payload
- `task_executions.metadata` ← EntryRequest.metadata

> 具体字段名由 Mike 在 A2_Schema_Registry 中最终确认，这里为 FSD 层建议映射。

---

## 5. 约束与不变性 (Invariants)

1. **entry_type 一旦写入，不再修改**：所有后续逻辑只能在该类型下工作。
2. **raw_query 模式下，payload.intent_ir 只能由 Intent Analyzer 写入**，其他组件不得篡改。
3. **EntryRequest 不承载 capability_chain**：任务分解完全属于 Router 侧（见 08.2）。
4. **metadata 永远不参与业务决策**，仅用于审计与排障。
