# 02.4 跨层失败处理

> **用途**：定义层间失败如何传递、谁负责处理

---

## 一、失败传递原则

- **Router 不修复错误**：捕获 Capability 失败，原样写入 audit，传递给上层
- **Entry 不掩盖**：Reject 即 Reject，不伪装成「成功但无结果」
- **Capability 必须声明失败**：status=failure 时必填 error、failure_mode

---

## 二、跨层场景

| 场景 | 行为 |
|------|------|
| Entry Reject | 不创建 Execution，返回 Reject 话术，记 Trace Log |
| Capability 超时 | Router 捕获，status=failed，audit 写入 timeout |
| Capability 缺 Memory | Capability 返回 failure，Router 写入 audit，不重试（除非 Policy 定义） |
| Evolution Recommendation 被拒 | Router 在 audit 中记录拒绝原因 |

---

## 三、不传递的边界

- **Soul Reject**：在 Entry/Intent/Content 层即中止，不进入 Capability
- **pending_confirmation 超时**：Orchestration 更新 status=expired，不触发 Router

## 四、跨面降级约束（2026-02-21 新增）

**业务查询不得静默降级**：当用户请求业务数据（金库、执行明细、市场概况等）时，若 Business Data Plane 无数据，必须明确返回澄清或「暂无该数据」，**禁止**静默降级为仅提供 Governance Data Plane（task_executions、audit_logs）作答。
