# 02.5 Capability 定义与接入

> **用途**：本 Module 内**唯一**对「Capability 是什么、有哪些、如何接入」的正式产品定义与核对依据。  
> **开发/接入 Capability 时**：以本文 + `FSD/02_Layer_Model/Capability_Contracts/` 为**产品开发核对的唯一基础**。  
> **母版**：定义与列表继承 `FSD/MAXshot_产品架构设计_v5.0_Autonomous_Soul.md` §2.5；v5.1 架构继承并引用。

---

## 一、Capability 定义

| 概念 | 定义 |
|------|------|
| **capability** | 被认证、可编排的**产品能力**，Runtime 唯一公民；仅 Router 按 Registry allowlist 调用。 |
| **skill** | 实现来源（代码 / Prompt / Workflow），是 Capability 的**内部填充物**；Runtime 以 Capability 为边界，不直接暴露 skill。 |
| **tool** | 研发辅助工具，Runtime 不可见。 |

**约束（本版必须遵守）**：

- Capability **不直接访问 DB**，数据仅通过 Router/context（含 `memory_refs`）传递（见 02.1、决议 7）。
- 运行时**只允许 Router 调用 Registry 中 allowlisted 的 capability**；任何 skill 不得绕过 capability 直连运行时。

### Intent ↔ Capability 绑定映射（强制同步）

> **这是「隐式耦合」的显式化**。每新增一个 Capability，下表必须同步扩展一行；否则 Intent Analyzer 无法将用户的自然语言路由到新能力。

| intent_name（Intent Analyzer 输出） | 路由到 capability_id | 用户语言示例 |
|------------------------------------|---------------------|------------|
| `ops_query` | `capability.data_fact_query` | 「金库几个？」「APY 多少？」 |
| `documentation` | `capability.product_doc_qna` | 「APY 怎么算的？」「产品逻辑？」 |
| `marketing_gen` | `capability.content_generator` | 「写条推文」「内容复盘」 |
| `small_talk` | （拒绝，友好告知超出范围） | 「你好」「天气怎么样」 |
| `out_of_scope` | （拒绝，引导至正确渠道） | 「帮我转 ETH」「预测价格」 |
| `unknown` | （澄清，最多 3 轮后拒绝） | 意图不明的模糊输入 |

**同步责任**：
- **新增 Capability 时**：LEO 写 CC → John/Alex 在 06.4 intent_schema 加对应 intent → Sam 在 Registry 代码加 handler → LEO 更新本表 + 00.6 §四。
- **缺任何一步 = 系统断裂**：详见 `00.6_Product_Experience_Map.md` §七 隐式耦合清单。

**扩展**：开发 3～10 个 capability 做接入时，新增能力须满足 §四 Registry 门槛，并在 `Capability_Contracts/` 下提供对应契约（CC_xx）。

## 三、Capability Registry 门槛 (MVP-6)

接入或新增 Capability 须满足以下**硬门槛**（不复杂但够用）：

| # | 门槛 | 说明 |
|---|------|------|
| 1 | 唯一 ID/Version | `capability_id` + `capability_version`，可升级/回滚。 |
| 2 | 统一 Envelope (Input/Output) | CapabilityInput / CapabilityOutput，见 Integration Spec、10_Appendix/A2。 |
| 3 | Evidence 结构 | 输出含 evidence（sources、sql_refs、doc_quotes 等），结构可空但必须存在。 |
| 4 | Audit 审计字段 | 含 `used_skills`、execution_id、capability_id、capability_version、status、failure_mode、invocation_source；见 09.1、09.2。 |
| 5 | Risk Policy | Side Effect 必须确认（pending_confirmation）；见 05.4。 |
| 6 | Failure Modes | 明确失败语义，写入 audit；可回放、可追溯。 |
