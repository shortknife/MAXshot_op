# 01.3 出口类型与系统行为

> **用途**：定义请求的终态与对应系统行为

---

## 一、出口类型

| 出口类型 | 含义 | 系统行为 |
|----------|------|----------|
| **completed** | 执行成功 | Execution status=completed，Audit 完整 |
| **failed** | Capability 执行失败 | Execution status=failed，audit 含 error |
| **rejected** | 用户拒绝确认 | Execution status=rejected，不触发 Router |
| **expired** | 确认超时 | Execution status=expired，不触发 Router |
| **user_abandoned** | 用户放弃澄清 | 记录审计，不创建 Execution |
| **unknown** | 轮次上限仍无法确定意图 | 记录审计，转人工或标记 unknown |
| **entry_reject** | Entry Gate 拒绝 | 不创建 Task/Execution |
| **soul_reject** | Soul 边界拒绝 | 不创建 Execution |

---

## 二、审计要求

所有出口（含 Reject）必须可追溯：
- 出口类型
- 时间戳
- 原因（若有）
- 关联的 entry_type、requester_id（若有）

---

## 三、与 01.2 的对应

- Entry Reject → exit_type=entry_reject
- Capability Reject → exit_type=failed（Execution 已存在）
- Soul Reject → exit_type=soul_reject

---

## 四、产品决议（本版唯一执行）

**决议 4（Output 以出口类型为一等公民）**  
- Output 层输入包含 **exit_type**（§一 八种）；按 channel 仅为同一 exit_type 下的展示差异。  
- 每种 exit_type 有对应话术/行为（最小集：completed、failed、entry_reject、rejected）；所有出口（含 Reject）须可追溯、符合 §二 审计要求。  
- **开发/验收要点**：设计「统一 Output」或开 Output 相关工票时，须包含 exit_type 与拒绝路径话术；不得仅做「成功结果按 channel 格式化」。
