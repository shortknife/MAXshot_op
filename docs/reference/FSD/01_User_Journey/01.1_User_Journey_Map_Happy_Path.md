# 01.1 用户路径：Happy Path

> **用途**：描述标准成功路径，作为 Rejection Path 的对照

---

## 〇、主路径：业务查询（2026-02-21 新增）

```
用户自然语言提问（金库、执行明细、市场概况等）
  → Entry Adapter 接收
  → Intent 分类（business_query / governance_action / content_operation）
  → 业务查询优先路由至 Business RPC/Query Capability
  → 业务数据查询（RPC + 结构化查询 + RAG）
  → 返回答案 + evidence（source RPC/table/evidence）
```

**原则**：业务查询不得静默降级为仅提供治理数据（task_executions、audit_logs）。

---

## 一、raw_query 路径

```
用户输入 (TG Bot / Admin OS Chat)
  → Entry Adapter 接收
  → Context Load（session_context snapshot）
  → Intent Analyzer（Soul 滤镜通过）
  → Sealer 写入 tasks + task_executions
  → (若 side_effect) pending_confirmation → 用户确认
  → Router 接收 execution_id
  → Task Decomposition + Working Mind Synthesis
  → Capability Chain 执行
  → 结果返回 + Audit 写入
```

---

## 二、structured 路径

```
Admin OS / Cron / Notion / System 输入
  → Entry Adapter 接收
  → Sealer 直接写入（跳过 Intent Analyzer）
  → Router 接收 execution_id
  → 执行
  → Audit 写入
```

---

## 三、timeline 路径

```
publishing_timeline 到期扫描
  → 防重锁（scheduled → publishing）
  → 到期校验（scheduled_at <= now()）
  → 封印 / 分发执行
  → Audit 写入
```

---

## 四、成功终态

- **Task**：active（scheduled）或 completed（ad_hoc）
- **Execution**：completed
- **Audit**：capability_chain、memory_refs、每步 status 已写入

---

## 五、产品决议（本版唯一执行）

**决议 3（三入口形态优先）**  
- **entry_type 为第一维度**：Entry 先定 `entry_type = raw_query | structured | timeline`，再按此分支（是否走 Intent、是否封印、防重锁）。  
- **渠道 → 默认形态**：TG 文本→raw_query；Notion webhook→structured；timeline 扫描→timeline。TG/Notion/Admin OS 为 **entry_channel**，仅审计与 Output 适配。  
- **开发/验收要点**：实现 Entry/总编排时必须有 entry_type 分支逻辑；文档保留「渠道→默认形态」对照表。
