# 05.4 Require Confirmation 流程

> **用途**：side_effect 操作的确认机制

---

## 一、触发条件

- Capability 声明 `risk_class=side_effect`
- Policy 指定 `require_confirmation=true`
- 或进入 Grey Area（见 05.3）

---

## 二、流程

1. 创建 Execution，status=pending_confirmation
2. 生成 confirmation_request（含 Preview）
3. **确认对象**（用户/管理员/外部系统，见 05.3 §3.1）确认 / 拒绝 / 超时
4. 确认 → status=confirmed，触发 Router
5. 拒绝/超时 → status=rejected/expired，不触发 Router

---

## 三、审计

confirmation_request 与 confirmation_result 必须写入 audit。

---

## 四、与 05.3 产品级闭环的衔接

- 确认对象由 Policy 显式定义，见 05.3 §3.1
- 同一 Grey 不自动学习，见 05.3 §3.2
- Confirmation 结果可经 Evolution 归因进入 Experience，见 05.3 §3.3

---

## 五、产品决议（本版唯一执行）

**决议 1（Side Effect 必须确认）**  
- **Phase 1 仅 Admin OS 确认**；TG/Notion 内确认列为后续迭代；契约与设计留扩展位。  
- 发布类（Publisher）等 Side Effect：先写 task_executions 且 status=pending_confirmation，**用户仅在 Admin OS 确认**后才触发 Router/执行发布。  
- **开发/验收要点**：开票或实现「发推/发布」时，必须体现「先 pending_confirmation → Admin OS 确认 → 再执行」；不得实现「TG 直接确认即发」除非明确列为后续迭代。
