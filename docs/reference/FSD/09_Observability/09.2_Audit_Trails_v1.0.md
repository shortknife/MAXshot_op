## 一、贯穿原则

- **execution_id** 是 Audit Trail 的贯穿键；单次执行从 Entry（或 Sealer 创建 Execution）到 Capability 完成，所有审计记录必须可经 execution_id 串联。
- **记录位置**：task_executions 表（状态、终态）、audit_logs 或等价存储（每步 Capability audit、Router 决策、Entry Gate 结果）。
- **不可变**：Execution 启动后的 Snapshot 不因后续表更新而改变；审计记录只追加、不篡改（见 03.3）。

---

## 二、记录位置与内容（2026-02-21 补充：答案溯源）

**客户面业务答案**：必须记录 **answer evidence provenance**（source RPC/table/evidence），供可解释性与审计追溯。

| 阶段 | 记录位置 | 必含内容 |
|------|----------|----------|
| **Entry / Gate** | Trace Log 或 task_executions 扩展 | entry_type、requester_id、gate_result（pass/reject/continue_chat）、reason（若 Reject） |
| **Sealer** | task_executions | task_id、execution_id、status=created、entry_type、intent_name、reason_for_pending（若 pending_confirmation） |
| **Router** | audit_logs / Exec Log | execution_id、capability_chain、memory_refs_ref、Intent Log 摘要 |
| **Capability** | CapabilityOutput.audit → audit_logs | execution_id、capability_id、capability_version、status、failure_mode、elapsed_ms、invocation_source |
| **终态** | task_executions | status=completed/failed/rejected/expired、timestamp |

---

## 三、与 Integration Spec 一致

- **CapabilityOutput.audit** 必须包含 execution_id、capability_id、capability_version、status、failure_mode（或 null）；见 Integration Spec 2.3、8。
- **pending_confirmation**：confirmation_request 与 confirmation_result 必须写入 audit；见 FSD 05.4、Integration Spec 5.2。
- **Evolution Recommendation**：Router 采纳或拒绝建议须在 Audit Log 中记录原因；见 Integration Spec 4.3、FSD 04.5。

---

## 四、产品决议（本版唯一执行）

**决议 6（业务数据优先）**：客户面业务答案必须来自 Business Data Plane；**治理表（task_executions、audit_logs）不参与主答案**；审计记录答案溯源（source RPC/table/evidence），不得用治理遥测冒充业务答案。  

**决议 8（Session 留痕）**：审计与 execution 关联 **session_id**（或等价）；留痕可串起「同一轮对话」；§二 记录位置与内容在实现时须含 session_id 传递与落库。  

**开发/验收要点**：验收含「不得用治理表当业务答案」的负面用例；新入口/留痕设计检查 session_id 是否写入审计。
