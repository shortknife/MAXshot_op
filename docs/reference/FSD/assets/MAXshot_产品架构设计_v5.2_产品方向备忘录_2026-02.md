## 🪞一、产品哲学：从企业工具观到需求体 Agent 观

### 1.1 为什么这件事必须先写清

如果这层哲学不先写清，后面的开发会不断滑回传统软件路径：

- 把产品做成一个"企业查询工具"
- 把用户当成一个"发起请求的账号"
- 把 AI 当成一个"聊天界面 + 若干自动化脚本"

这不是我们真正要做的东西。

MAXshot 进入 v5.2 后，必须明确一件事：

> **我们不是在做一个会聊天的企业工具，而是在做一个面向"需求体"的 Agent 产品。**

这里的"需求体"，不是企业，也不是个人，而是一个完整的、持续存在的真实需求单元。
它可能来自一个创始人、一个运营同学、一个小团队，甚至是一组长期重复出现的业务意图。
它有目标、有偏好、有历史、有约束、有上下文，也有尚未说出口但真实存在的潜在需求。

产品面对的，不该只是"一句 query"，而是这个 query 背后的整个需求体。

### 1.2 什么是我们要服务的"需求体"

| 传统工具视角 | v5.2 Agent 视角 |
|--------------|----------------|
| 用户是账号 / 企业席位 | 用户是持续存在的需求体 |
| 输入是一句话 | 输入是当前问题 + 历史上下文 + 目标 +偏好 +约束 |
| 产品负责回答 | 产品负责理解、组织、回答、解释、记忆、必要时建议 |
| 只追求功能正确 | 同时追求可控、可实现、真实、稳定、有温度、有想象力 |

**需求体**这个定义，决定了 MAXshot 不是一个单轮问答器，而是一个可持续服务的 Agent。

### 1.3 v5.2 的产品定义

MAXshot v5.2 应被定义为：

> **一个面向需求体的 Agent Operating System。**

它的核心任务不是聊天，而是把生成力真正变成可交付的能力。

所以它必须同时满足六个要求：

| 要求 | 说明 |
|------|------|
| **可控** | 行为边界清楚，能被约束、能被审计、能被回放 |
| **可实现** | 不是空泛陪聊，而是能调动真实能力，交付真实结果 |
| **真实** | 数据有来源，结论可解释，不伪造事实 |
| **稳定** | 行为一致、故障可降级、关键链路可验证 |
| **有记忆** | 记得住用户、历史、偏好、长期目标 |
| **有温度与想象力** | 不只是复述结果，还能理解动机、提供洞察、提出可能性 |

### 1.4 这六条就是后续检验准绳

从 v5.2 开始，MAXshot 后续所有功能、Prompt、Workflow、UI、回答风格，都应该回到这六条来验收：

1. **它可控吗？**
2. **它真的能实现结果吗？**
3. **它基于真实世界和真实数据吗？**
4. **它在不确定场景下仍然稳定吗？**
5. **它记住了这个需求体，而不是每次都像第一次见面吗？**
6. **它有温度和想象力，能把答案升级成帮助吗？**

如果一个能力只满足"能跑通"，但不满足上面至少四条，它就还不是我们要的 Agent 能力。

### 1.5 对开发团队的直接约束

这套哲学不是写给未来看的口号，而是从现在开始约束我们的开发判断：

| 开发问题 | v5.2 的判断方式 |
|----------|----------------|
| 要不要继续堆 regex / 模板？ | 仅作为短期兜底，不作为长期主方案 |
| 要不要让 LLM 参与理解与解释？ | 要，且应成为主路径之一 |
| 要不要做记忆？ | 要，没有记忆就不是 Agent |
| 要不要做解释与建议？ | 要，只有结果没有解释，仍是工具而不是助手 |
| 要不要保留确定性架构？ | 要，确定性是守卫层，不是对 LLM 的否定 |

一句话说：

> **v5.2 不是让 MAXshot 更像企业软件，而是让 MAXshot 更像一个可信、稳定、真实、有温度的 Agent。**

### 1.6 v5.2 产品验收标准（Agent 版）

以下表格把「六条准绳」落成可执行验收项。**从 v5.2 起，功能/需求验收时除了「做没做完」，还要用这张表自检是否还在准绳内。**

| 准绳 | 验收问题 | 通过标准（示例） |
|------|----------|------------------|
| **可控** | 行为边界是否清楚？能否审计、回放？ | 有明确 capability 边界；关键决策有 audit/trace；可配置开关或降级 |
| **可实现** | 是否调用真实能力、交付真实结果？ | 不只有话术，有真实数据/执行/写回；失败有明确错误与降级路径 |
| **真实** | 数据是否有来源？结论是否可解释？ | 数据来自业务表/RPC/已知来源；回答可追溯到 SQL/证据；不伪造数字 |
| **稳定** | 不确定场景下是否仍可验证、可降级？ | 有 fallback；关键路径有校验；异常有日志与可复现 |
| **有记忆** | 是否记住需求体，而非每次都像第一次？ | 跨 session 有 User Profile/偏好/历史；回答能利用上下文 |
| **有温度与想象力** | 是否把答案升级成帮助（解释、洞察、建议）？ | 有简短解释或「你可能还想问」；在合适场景有策略建议或主动提示 |

**判定规则**：

- 新功能 / 大改版：六条中至少满足 **四条** 才算在 Agent 准绳内；不满足的需有明确改进计划。
- 小修小补 / 热修：可不逐条过，但不得削弱已有「可控 / 真实 / 稳定」表现。
- 产品与研发评审时，用本表做「是否偏离 v5.2 定位」的快速检查。

## 🧬 二、核心哲学转变：从 Code-First 到 LLM-First, Code-Guard

### 1.1 问题观察

v5.1 开发过程中，我们发现一个深层矛盾：

| 层面 | v5.1 现状 | 问题 |
|------|-----------|------|
| **Intent 识别** | 正则模板匹配（`VAULT_REQUIRED_PATTERNS`、`GLOBAL_OR_TOP_N_APY`） | 每新增一种问法就要加正则，永远追不完 |
| **SQL 生成** | Tier1 模板优先，Tier2 LLM 是 fallback | 模板覆盖率有限，长尾问题无法预设 |
| **Slot 提取** | 硬编码 clarification prompt | 用户体验僵硬，无法动态适应不同问法 |
| **回答生成** | 数据 + 固定格式化 | 缺少解释、缺少洞察、像机器不像助手 |

**根因**：团队惯性地用「写代码」解决「自然语言理解」问题。这在传统软件中是正确的，但在 AI 产品中是**反直觉的低效**。

### 1.2 哲学重定义

```
v5.1: Router(确定性) → 代码模板(确定性) → LLM(不可信, fallback)
v5.2: Router(确定性) → LLM(理解/生成) → LLM(验证/确定性) → Code(兜底/审计)
```

**核心命题**：

> **确定性不是通过「不用 LLM」来实现的，而是通过「用代码编排和验证 LLM」来实现的。**

| 原则 | v5.1 解读 | v5.2 重定义 |
|------|-----------|------------|
| **Router = 确定性调度器** | 不变 | **不变**。Router 仍用代码，仍是确定性心脏 |
| **LLM = 不可信建议源** | LLM 只在 Tier2/长尾时使用 | **LLM 是主要理解/生成引擎**，但「不可信」约束通过双重验证实现 |
| **No Trace = Bug** | 不变 | **不变**。所有 LLM 调用必须有 trace |
| **Execution = 不可变历史** | 不变 | **不变** |

**新增原则**：

| 新原则 | 说明 |
|--------|------|
| **LLM-First, Code-Guard** | 能用 LLM 解决的理解/生成/解释问题，优先用 LLM；代码负责编排、验证、兜底 |
| **双重 LLM 验证 = 新的确定性** | 一个 LLM 输出，另一个 LLM 验证一致性；通过「交叉验证」替代「模板硬编码」的确定性 |
| **Agent Thinking** | 产品不是「查询工具」，是「会思考的助手」；每个回答背后有推理链 |
| **温度即产品** | 回答不只是数据，要有解释、有洞察、有温度 |

### 1.3 与架构宪法的关系

**不违反宪法**。宪法说的是：
- Router = 确定性调度器 ✅（v5.2 Router 仍是代码）
- LLM = 不可信建议源 ✅（v5.2 通过双重验证约束 LLM）
- Router 不能让 LLM 直接调度 ✅（v5.2 LLM 仍只做建议，Router 做最终决策）

**v5.2 拓展的是**：在 Router 之下的 Capability 执行层，从「代码模板为主」变为「LLM 理解为主 + 代码验证」。

## 🏗️ 三、架构演进：LLM-First Capability Layer

### 2.1 总体架构对比

```
v5.1 架构:
  Entry → Router(Code) → Capability(Code Templates + LLM Fallback) → Result

v5.2 架构:
  Entry → Router(Code) → Capability(LLM Understanding + LLM Verification + Code Guard) → Result + Insight
```

### 2.2 Intent 层演进

| 维度 | v5.1 | v5.2 |
|------|------|------|
| **识别方式** | 规则层（regex）+ LLM 层（fallback） | **LLM 为主**，schema_index 作为 LLM 的上下文注入 |
| **Slot 提取** | 硬编码 pattern + 固定 clarification | **LLM 动态引导**：根据 schema_index 知道需要什么 slot，用自然语言追问 |
| **确定性保障** | 代码模板 | **第二个 LLM 验证 Intent 分类结果的一致性**；Router 校验 intent 在注册列表中 |
| **扩展性** | 每种新问法加正则 | **零代码扩展**：只需更新 schema_index（表/列/描述），LLM 自动理解新问法 |

**设计细节**：

```
用户: "哪个金库收益最高？"

v5.1 流程:
  1. regex 匹配 → VAULT_REQUIRED_PATTERNS → needs_clarification=true → "请指定金库名称"
  （用户体验差：用户明明在问"最高的是哪个"，系统却要求指定具体名称）

v5.2 流程:
  1. LLM-A (Intent Analyzer) 接收 query + schema_index
     → intent: data_fact_query
     → slots: { metric: "avg_net_apy", aggregation: "max", scope: "all_vaults" }
     → needs_clarification: false（LLM 理解「最高」= 全局排序，无需指定金库）
  2. LLM-B (Verifier) 验证: intent 与 slots 是否自洽？schema_index 中是否有对应表/列？
     → verified: true
  3. Router 校验: intent 在注册列表中？slots 类型合法？
     → pass
  4. SQL 生成 → 执行 → 回答
```

### 2.3 SQL Engine 演进

| 维度 | v5.1 | v5.2 |
|------|------|------|
| **主路径** | Tier1 模板优先 | **LLM + schema_index 生成为主**（原 Tier2 升级为主路径） |
| **Tier1 定位** | 主力 | **性能缓存层**：高频 query 的 LLM 生成结果自动沉淀为 Tier1，减少 LLM 调用 |
| **确定性保障** | 模板本身确定 | **双重验证**：LLM-A 生成 SQL → LLM-B 审查 SQL 合法性/安全性 → Code 白名单校验 |
| **沉淀机制** | Tier3 手动审核 | **自动沉淀**：同一 intent+slots 组合调用 ≥ N 次且结果一致 → 自动缓存为 Tier1 |

```
v5.2 SQL Engine 分层重定义:

Tier-Cache (原 Tier1): 缓存层
  - 来源：LLM 生成的 SQL 经验证+沉淀后的缓存
  - 命中时直接执行，跳过 LLM 调用（降本提速）
  - 自动沉淀 + 人工审核可选

Tier-LLM (原 Tier2，升级为主路径): LLM 生成层
  - 输入：user query + schema_index + intent/slots
  - LLM-A 生成 SQL
  - LLM-B 验证 SQL（表/列白名单、只读约束、逻辑合理性）
  - Code Guard 最终校验

Tier-Fallback: 兜底层
  - LLM 生成失败或验证不过时的人类可读错误
  - 记录失败案例，用于后续优化
```

### 2.4 Answer 层演进

| 维度 | v5.1 | v5.2 |
|------|------|------|
| **回答内容** | 数据 + 格式化 | **数据 + 解释 + 洞察 + 建议** |
| **解释** | 无 | LLM 解释「为什么是这个值」（如：该金库 APY 最高是因为 XXX 策略） |
| **洞察** | 无 | RAG 语义库匹配相关上下文，猜测用户问这个问题的深层原因 |
| **建议** | 无 | 基于数据趋势和用户历史，主动给出策略建议（如：这个金库 APY 在下降，建议关注 XXX） |

## 🤖 四、从被动查询到主动 Agent

### 3.1 产品人格：智能管家

> 把产品当做一个万能的管家。当主人来问的时候，我们用拟人的思路回答他。

| 维度 | v5.1 | v5.2 |
|------|------|------|
| **角色** | 查询工具 | **智能管家** |
| **交互模式** | 问→答 | **问→理解→答→解释→建议→跟进** |
| **主动性** | 被动等待 | **主动推送**（数据异常、趋势变化、策略建议） |
| **记忆** | 无跨 session 记忆 | **跨 session 记忆**：记住用户关注的金库、常问的问题、偏好的回答风格 |
| **人格** | 无 | **可配置的管家人格**：专业但有温度，简洁但有洞察 |

### 3.2 管家的六种能力

| 能力 | 说明 | 实现方式 |
|------|------|----------|
| **①理解** | 理解用户的真实意图，不只是字面意思 | LLM Intent Analyzer + 上下文记忆 |
| **②追问** | 信息不足时，用自然语言智能追问（非硬编码模板） | LLM 引导对话，3-5 轮后用默认值兜底 |
| **③回答** | 精准回答 + 数据证据 | schema_index 驱动的 SQL + 执行结果 |
| **④解释** | 解释「为什么是这个值」 | LLM 结合数据和业务知识生成解释 |
| **⑤洞察** | 猜测用户问这个问题的深层原因，给出相关信息 | RAG 语义库匹配 + LLM 推理 |
| **⑥建议** | 基于数据趋势和用户历史，主动给策略建议 | Memory + Evolution 简化版 |

### 3.3 对话示例

```
用户: "金库收益最高的是哪个？"

v5.1 回答:
  "请指定金库名称。"（clarification，因为 regex 识别为需要指定金库）

v5.2 管家回答:
  "📊 当前收益最高的金库是 **Morpho-ETH-USDC**，24h 平均 APY 为 12.3%。

  💡 补充说明：
  - 这个 APY 较上周上涨了 2.1%，主要因为近期 ETH 借贷需求增加
  - 排名第二的 Native-WBTC 为 10.8%，差距在 1.5% 以内

  🎯 你可能想知道：
  - 「Morpho-ETH-USDC 的 APY 趋势如何？」
  - 「按照这个趋势，我们的收益策略是否需要调整？」"
```

### 3.4 主动推送场景

| 场景 | 触发条件 | 推送内容 |
|------|----------|----------|
| **APY 异常波动** | 某金库 APY 24h 变化 > 30% | "⚠️ Morpho-ETH-USDC APY 从 12.3% 降至 8.1%，降幅 34%。建议关注。" |
| **策略建议** | 用户连续 3 次查询某金库 | "💡 你最近很关注 Morpho-ETH-USDC，要不要我帮你设置一个日报？" |
| **数据摘要** | 每日/每周定时 | "📋 本周金库数据摘要：总锁仓 +5.2%，最优 APY 金库变更为 XXX" |
| **风险预警** | TVL 大幅下降或异常交易 | "🔴 Native-WBTC TVL 在过去 2h 下降 15%，请注意风险。" |

## 🧠 五、记忆系统：从无状态到有温度

### 4.1 借鉴 OpenClaw 的记忆哲学

OpenClaw 的双层记忆（日常 + 长期）给了我们启发：一个好的 Agent 不只是能回答问题，更要**记住用户**。

### 4.2 v5.2 记忆层设计

| 层次 | 说明 | 存储 | 生命周期 |
|------|------|------|----------|
| **Session Memory** | 当前对话上下文 | 内存 / Redis | 对话结束后压缩归档 |
| **User Profile** | 用户偏好、关注点、常用查询 | Supabase | 长期 |
| **Business Memory** | 数据趋势、历史洞察、策略记录 | Supabase + RAG 向量库 | 长期 |
| **Interaction Pattern** | 用户交互模式（问什么、什么时间问、关注什么指标） | Supabase | 定期聚合 |

### 4.3 记忆驱动的个性化

```
场景：用户 Alex 每天早上 9 点查金库数据

v5.1: 每次都像第一次 — 没有记忆
v5.2:
  - Session Memory: 记住今天对话中已经问过什么
  - User Profile: 知道 Alex 最关注 Morpho-ETH-USDC 和 Native-WBTC
  - Business Memory: 知道上周 Morpho-ETH-USDC 的 APY 趋势
  - Interaction Pattern: 知道 Alex 习惯在早上查数据，偏好简洁回答

  管家主动打招呼:
  "☀️ 早上好 Alex！今日金库快报：
  - Morpho-ETH-USDC APY 12.3%（↑0.5%）
  - Native-WBTC APY 10.8%（持平）
  你上次提到的收益策略调整评估还需要看吗？"
```

## 🔄 六、双重 LLM 验证：新的确定性范式

### 5.1 为什么需要双重验证

| 方案 | 确定性来源 | 问题 |
|------|-----------|------|
| **代码模板** | 代码本身确定 | 覆盖率有限，维护成本高，体验僵硬 |
| **单 LLM** | 无确定性保障 | 幻觉、不一致、不可预测 |
| **双 LLM 验证** | 交叉验证 + 代码兜底 | ✅ 覆盖率高 + 确定性可控 + 体验自然 |

### 5.2 验证架构

```
                    ┌─────────────────────────┐
                    │    Router (Code)         │
                    │    确定性调度，不变       │
                    └────────────┬────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │    LLM-A (Worker)        │
                    │    理解 / 生成 / 解释     │
                    │    输出: intent, slots,   │
                    │          SQL, answer      │
                    └────────────┬────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │    LLM-B (Verifier)      │
                    │    验证一致性 / 合理性    │
                    │    检查: intent 与 query  │
                    │    匹配？SQL 与 schema    │
                    │    一致？answer 与 data   │
                    │    相符？                 │
                    └────────────┬────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │    Code Guard            │
                    │    最终校验 + 审计留痕    │
                    │    白名单 / 只读 / trace  │
                    └─────────────────────────┘
```

### 5.3 验证维度

| 阶段 | LLM-A 输出 | LLM-B 验证内容 | Code Guard |
|------|-----------|---------------|------------|
| **Intent** | intent_name + slots | 「这个 intent 和用户原文语义一致吗？」 | intent 在注册列表中？ |
| **SQL** | SELECT ... | 「SQL 引用的表/列在 schema_index 中吗？语义与 intent 一致吗？有无注入风险？」 | 白名单表/列校验、只读约束 |
| **Answer** | 自然语言回答 | 「回答中的数字与 SQL 执行结果一致吗？有无凭空编造的数据？」 | 数据引用完整性 |

### 5.4 成本控制

| 策略 | 说明 |
|------|------|
| **Tier-Cache** | 高频 query 沉淀后跳过双 LLM 调用 |
| **Worker/Verifier 模型分级** | LLM-A 用强模型（如 GPT-4/Claude），LLM-B 用快速模型（如 GPT-4o-mini） |
| **置信度短路** | 如果 LLM-A 输出 confidence > 0.95 且命中 Tier-Cache 近似 pattern，可跳过 LLM-B |
| **批量验证** | 对于非实时场景（如日报生成），可批量验证 |

## 🌊 七、n8n 编排 — 回归编排本质

### 6.1 n8n 的重新定位

v5.1 的「轻 n8n」策略把大量逻辑移到代码中。v5.2 重新审视 n8n 的价值：

| 维度 | v5.1 | v5.2 |
|------|------|------|
| **n8n 定位** | 总编排 + Prompt 提取 + LLM 节点 | **Agent 思维链编排器**：每个 Capability 是一条 n8n 工作流，可视化展示 Agent 的「思考过程」 |
| **代码定位** | 业务逻辑主体 | **Guard / Validator / Executor**：校验、执行、审计 |
| **可视化价值** | 流程图 | **Agent 思维链可视化**：Intent → Slot → SQL → Verify → Answer → Insight，每步可追溯 |

### 6.2 一条 query 的 n8n 编排

```
[Entry] → [Context Load] → [LLM-A: Intent+Slot] → [LLM-B: Verify Intent]
    → [LLM-A: SQL Gen] → [Code: SQL Whitelist Check] → [LLM-B: Verify SQL]
    → [Code: Execute SQL] → [LLM-A: Answer+Explain+Insight]
    → [LLM-B: Verify Answer] → [Code: Trace+Audit] → [Sealer]
```

每个节点可独立替换、调试、观测。这就是 n8n 作为编排器的真正价值。

## 📐 八、LEO 的更多想法 + 开放讨论

### 7.1 Confidence-Driven Routing（置信度驱动路由）

> 不只是「匹配/不匹配」的二分法，而是用 LLM 的 confidence score 驱动路由决策。

```
LLM-A 输出 confidence:
  ≥ 0.9  → 直接执行，跳过部分验证
  0.7-0.9 → 正常双 LLM 验证流程
  0.5-0.7 → 主动追问用户确认
  < 0.5  → 明确告知「我无法理解这个问题」
```

比硬编码 `needs_clarification` 更自然、更精准。

### 7.2 Progressive Disclosure（渐进式披露）

回答策略：先给最核心的答案，然后提供「展开」选项。

```
管家: "Morpho-ETH-USDC 当前 APY 12.3%，是所有金库中最高的。"
     [📊 查看详细数据] [📈 查看趋势] [💡 查看策略建议]
```

TG Bot 可通过 Inline Keyboard 实现；Admin OS 可通过可折叠面板实现。

### 7.3 Learning Loop（学习闭环）

```
用户新问法 → LLM 理解 → 双重验证 → 执行成功
    ↓
  自动记录 (query_pattern, intent, slots, SQL)
    ↓
  同一 pattern 出现 ≥ N 次且结果一致
    ↓
  自动沉淀为 Tier-Cache（可选人工审核）
    ↓
  下次命中 Cache，跳过 LLM（降本提速）
```

系统越用越快、越用越准，这才是 AI 产品应有的飞轮效应。

### 7.4 Persona as a Service（人格即服务）

管家的「人格」应该是可配置的：

| 参数 | 示例值 | 说明 |
|------|--------|------|
| **communication_style** | professional / casual / concise | 沟通风格 |
| **proactivity_level** | low / medium / high | 主动推送频率 |
| **insight_depth** | data_only / with_explanation / with_strategy | 回答深度 |
| **language** | zh-CN / en | 语言偏好 |
| **focus_areas** | ["morpho", "native", "apy"] | 关注领域 |

存入 User Profile，管家行为随用户偏好调整。

### 7.5 从 Data Query 到 Strategy Agent（终极愿景）

```
Level 1 (v5.1): 被动查询 — "金库收益最高的是哪个？" → 返回数据
Level 2 (v5.2): 主动解释 — "最高的是 Morpho，因为 XXX，趋势是 YYY"
Level 3 (v5.2+): 主动建议 — "基于最近的 APY 趋势，建议调整收益策略 Prompt 的权重参数"
Level 4 (v6.0): 自主策略 — "我已经根据数据分析，准备了三套策略方案供你选择"
```

用户说的一句话启发了这个分级：

> "用户问了很多金库 APY，他真实的想法是，我如何根据这些数据来改变收益策略。从一个被动查询的产品提升到给主动提供策略的 Agent 产品。"

## 🗣️ 十、开放讨论点

以下是我认为需要团队讨论的核心问题：

### Q1: 双重 LLM 的成本可接受吗？
- 每次 query 需要 2-4 次 LLM 调用（Intent Worker + Verifier + SQL Worker + Verifier）
- Tier-Cache 沉淀后可显著降低调用频率
- 需要做成本估算：假设日均 100 次 query，LLM 成本 vs. 手动维护正则模板的人力成本

### Q2: Verifier LLM 的有效性如何保证？
- 如果 Verifier 也犯错怎么办？
- 建议：Verifier 不做开放式判断，只做 checklist 式校验（表/列存在？只读？数字一致？）
- Code Guard 作为最终防线

### Q3: 主动推送会不会成为骚扰？
- 需要用户可控的推送频率
- 初期保守：只推送 P0 级异常（APY 变化 > 30%、TVL 异常）
- 用户反馈后逐步放开

### Q4: 记忆系统的隐私边界？
- 记住用户偏好 vs. 隐私合规
- 建议：用户可查看/删除自己的 Profile
- 数据本地化存储（Supabase 自托管）

### Q5: 与 OpenClaw Growth Engine 的协同？
- v5.2 的管家能力是否与二黑团队的能力重叠？
- 建议：管家面向「Ops 查询 + 策略建议」，二黑面向「Marketing 执行」，通过数据层对接
