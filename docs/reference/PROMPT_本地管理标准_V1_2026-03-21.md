# Prompt 本地管理标准 V1

## 1. 目标

当前项目不再以 Supabase `prompt_library` 作为 Prompt 管理首选。  
本项目采用：

**本地文件真源 + 运行时按需接入**

目标有三个：

1. 让 Prompt 版本可审查、可对比、可回滚
2. 让 Prompt contract 与代码 contract 保持一致
3. 让每个 Step 的 Prompt 管理方式可复用，而不是散落在数据库或运行时代码里

### 2.2 一份 Prompt 必须有两层文件

每个 Prompt 必须同时存在：

1. **机器可读文件**
   - 用于运行时加载、测试、schema 校验
2. **说明文档**
   - 用于解释：
     - 为什么存在
     - 输入是什么
     - 输出 contract 是什么
     - 边界是什么

### 2.3 不允许的做法

不允许：
1. 只在数据库里维护 Prompt
2. 只在代码字符串里内嵌 Prompt
3. Prompt 变了，但没有同步文档和 contract
4. Prompt 输出 contract 与运行时 schema 不一致

## 4. Prompt 生命周期

每个 Prompt 都必须显式标明状态：

- `proposed`
- `approved`
- `active`
- `deprecated`
- `retired`

### 状态含义

#### `proposed`
- 仅设计完成
- 未接入 runtime

#### `approved`
- 架构与 contract 已评审通过
- 允许进入开发接入

#### `active`
- 已成为运行时主路径

#### `deprecated`
- 仍保留兼容，但不再推荐使用

#### `retired`
- 不再允许运行时使用，仅保留历史

## 5. Prompt 规格最小字段

每个 Prompt 至少要有这些字段：

```json
{
  "slug": "turn_relation_classifier_op_v1",
  "name": "Turn Relation Classifier",
  "status": "proposed",
  "model_config": {},
  "input_contract": {},
  "output_contract": {},
  "system_prompt": "...",
  "user_prompt_template": "...",
  "notes": []
}
```

必须明确：
1. `slug`
2. `status`
3. `model_config`
4. `input_contract`
5. `output_contract`
6. `system_prompt`
7. `user_prompt_template`

## 6. Prompt 与代码的映射规则

### 6.1 一个 Prompt 必须映射到一个明确模块

例如：
- `turn_relation_classifier_op_v1`
  -> `context-manager/turn-relation-classifier.ts`

### 6.2 一个 Prompt 必须映射到一个明确的 Step

例如：
- Step 2:
  - `turn_relation_classifier_op_v1`
- Step 3:
  - `intent_analyzer_op_v1`

### 6.3 Prompt 文档必须写清边界

例如：
- Step 2 的 Prompt 不得输出 capability 最终结论
- Step 3 的 Prompt 不得处理 session lifecycle

## 7. Step 2 对 Prompt 管理的具体要求

Step 2 后续至少会需要一类新 Prompt：

### `turn_relation_classifier_op_v1`
作用：
- 判断当前消息与前文关系

必须输出：
- `turn_relation.type`
- `confidence`
- `reason`

不得输出：
- `matched_capability_ids`
- `intent_type`
- `gate_result`

因为这些属于 Step 3 以后。

---

## 8. 文档更新规则

每新增一个正式 Prompt，必须同步更新：

1. 机器真源 JSON
2. 对应 Prompt Spec 文档
3. `docs/reference/README.md`
4. 对应 Step 实施计划中的 Prompt 章节

缺一项，就不算完成。

---

## 9. 评审规则

Prompt 进入 `approved` 前，至少要过这 4 项：

1. 输入 contract 是否明确
2. 输出 contract 是否可被代码严格消费
3. 与所属 Step 的边界是否一致
4. 是否已有测试样例和失败样例

---

## 10. 本标准的作用

本文之后，Prompt 管理不再是“写一个 Prompt 就结束”，而要经过：

1. 规格定义
2. 本地真源登记
3. 文档化
4. Step 对齐
5. 运行时接入
6. 测试矩阵验证

只有这样，后续每个 Step 才不会再次失衡。
