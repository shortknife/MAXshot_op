## 1. 背景与结论

MAXshot 当前主流程已明确为 9 步，其中 Step 2 的职责不应再被理解为“加载几个上下文字段”，而应被正式定义为：

**Conversation Context Manager（对话上下文管理层）**

它解决的核心问题只有四类：

1. 这条消息属于哪个 session / thread
2. 它与上文是什么关系
3. 当前允许下传给后续步骤的上下文是什么
4. 哪些上下文必须被隔离、清空或通过 memory 检索补回

本文对照了三类信息源：

1. **MAXshot 当前实现**
2. **OpenClaw（最新代码 / 文档）**
3. **Letta（最新代码 / 文档）**

结论先行：

- OpenClaw 最值得借鉴的是：**session/thread key 体系、session 生命周期、context compaction / pruning、gateway 作为 session truth source**
- Letta 最值得借鉴的是：**多层 memory 模型（core / recall / archival）以及“语义检索型历史回溯”**
- MAXshot Step 2 的正确方向应是：
  - **LLM 主导 turn relation 判断**
  - **Code 主导 session 装载、registry 装载、memory runtime、上下文继承策略、边界防护**
- 当前 MAXshot 实现只达到：
  - “规则驱动的业务查询续接 MVP”
  - 还不是正式的 Conversation Context Manager

## 3. Step 2 的标准目标（MAXshot 目标态）

### 3.1 Step 2 不负责什么
Step 2 不负责：
- capability 匹配
- intent 最终分类
- gate 决策
- router 调度
- capability 执行

这些属于 Step 3 以后。

### 3.2 Step 2 必须负责什么
Step 2 必须负责：

1. `session_resolution`
   - 这条输入属于哪个 session / thread
   - 是继续、fork 新分支、还是 reset

2. `turn_relation`
   - 当前话语和上文的关系：
     - `new_session`
     - `continuation`
     - `clarification_reply`
     - `correction`
     - `new_topic_same_window`
     - `history_callback`

3. `conversation_context`
   - pending clarification
   - active session context
   - 历史回溯入口

4. `registry_context`
   - 当前 active capability 集合

5. `memory_runtime`
   - 允许本轮使用哪些 memory refs
   - 来源策略是什么

6. `context_policy`
   - 哪些上下文允许继承
   - 哪些必须丢弃
   - 哪些必须触发历史检索

### 3.3 推荐输出 Contract
Step 2 推荐统一输出 `Context Envelope`：

```json
{
  "session_resolution": {
    "session_id": "session_xxx",
    "thread_action": "continue",
    "confidence": 0.95
  },
  "turn_relation": {
    "type": "continuation",
    "confidence": 0.91,
    "reason": "user narrows prior scope by chain"
  },
  "conversation_context": {
    "pending_clarification": {
      "exists": true,
      "scope": "yield",
      "missing_slots": ["time_window"]
    },
    "active_context": {
      "scope": "yield",
      "chain": null,
      "protocol": null,
      "vault_name": null,
      "time_window_days": 7,
      "aggregation": "avg"
    }
  },
  "registry_context": {
    "active_capability_ids": [
      "capability.data_fact_query",
      "capability.product_doc_qna",
      "capability.content_generator",
      "capability.context_assembler"
    ]
  },
  "memory_runtime": {
    "source_policy": "router_context_only",
    "ref_ids": [
      "capability_registry_v1",
      "capability_registry_v1:capability.data_fact_query"
    ],
    "memory_ref_count": 2
  },
  "effective_query": "那 arbitrum 呢？",
  "context_ready": true
}
```

## 5. OpenClaw 可借鉴点

### 5.1 核心观察
OpenClaw 最成熟的不是“意图识别”，而是：

- session key 设计
- channel / peer / thread 隔离
- session 生命周期
- transcript / compaction / pruning
- gateway 作为 source of truth

### 5.2 值得借鉴的设计点

#### A. Session 是正式对象，不是附加字段
OpenClaw 明确定义：
- direct chat
- group
- channel
- topic/thread
- cron/hook/node

都映射到稳定 `sessionKey`。

借鉴建议：
- MAXshot Step 2 应把 `session_id` / `thread_id` 正式建模
- 不要只把 session 当成 chat API 的一个辅助字段

#### B. Session key 的隔离粒度清晰
OpenClaw 文档明确了：
- DM 可以共享主 session，也可以按 peer/channel/account 隔离
- 组聊、频道、topic、thread 都有独立 key

借鉴建议：
- MAXshot 后续若扩 TG/Notion/Web 多入口，Step 2 应显式定义 session scope policy：
  - main
  - per-user
  - per-channel-user
  - per-thread

#### C. Gateway 是 session truth source
OpenClaw 明确要求：
- session state 由 gateway 管
- UI 不直接读本地 transcript 来拼 session 状态

借鉴建议：
- MAXshot 后续如果扩多客户端，不应让前端/各 channel 自己维护 session truth
- Step 2 需要一个中心 truth source

#### D. Transcript 与 session store 分层
OpenClaw 将：
- `sessions.json` 作为小型可变状态
- `*.jsonl` transcript 作为 append-only 历史

借鉴建议：
- MAXshot 后续应区分：
  - 轻量 session store
  - 完整 turn history / audit history
- 不要把全部状态都塞到一个会话 Map 里

#### E. Session pruning / compaction / memory flush 是正式机制
OpenClaw 把：
- session pruning
- compaction
- pre-compaction memory flush
都设计为标准生命周期的一部分。

借鉴建议：
- MAXshot Step 2 后续要考虑上下文溢出后的压缩策略
- 尤其是“多轮对话长了怎么办”不能只靠简单 TTL

### 5.3 对 MAXshot 最有直接价值的借鉴
直接可映射到 Step 2 的，是这 5 个：
1. session key taxonomy
2. gateway/source-of-truth 思想
3. session store vs transcript store 分层
4. pruning/compaction 生命周期
5. thread/topic 作为一级会话边界

### 5.4 不建议直接照搬的部分
1. OpenClaw 的大量 channel 细节
2. agent-to-agent session tool 全套机制
3. 其特定 gateway / CLI / broadcast groups 运行方式

这些对 MAXshot Step 2 不是当前主线。

## 6. Letta 可借鉴点

### 6.1 核心观察
Letta 的强项不在 session 路由，而在：

- 记忆分层
- 记忆读写作为 agent 原生能力
- recall / archival / core memory 的明确划分
- 历史回溯使用检索，不靠拼接上下文

### 6.2 值得借鉴的设计点

#### A. Core / Recall / Archival 三层 memory 模型
Letta 明确区分：
- `core memory`
  - 永远在上下文中的基础事实
- `recall memory`
  - 对话历史，可搜索
- `archival memory`
  - 更长期、更结构化、上下文外检索的持久记忆

借鉴建议：
- MAXshot Step 2 应避免把“所有记忆”混为一层 `memory_refs`
- 至少在 Contract 上区分：
  - active in-context state
  - searchable history
  - durable curated memory

#### B. 历史回溯用检索，不用硬拼提示
Letta 提供：
- `conversation_search`
- `archival_memory_search`
- `core_memory_append`
- `core_memory_replace`

借鉴建议：
- `history_callback` 不应实现成“把旧 query 文本硬拼回来”
- 应实现成：
  - 识别这是历史回溯
  - 再触发 recall search

#### C. Memory 是 Agent 的长期自我进化能力
Letta 的文档和系统 prompt 强调：
- 记忆是 agent 的自我演化基础
- 模型需要主动写 memory，而不是只依赖外部拼接

借鉴建议：
- MAXshot Step 2 虽然不直接负责 memory 编辑，但要为“可演化 agent”预留 memory runtime contract
- 不能把 Step 2 降格成一个 query patcher

#### D. Summarization / recall helper 机制
Letta 有专门的 summarizer / recall helper，把即将溢出的上下文总结成可检索记忆。

借鉴建议：
- MAXshot 后续可在 Step 2 邻近层引入：
  - conversation summary
  - branch summary
  - user/topic durable notes

### 6.3 对 MAXshot 最有直接价值的借鉴
最值得拿来改造 Step 2 的，是这 4 个：
1. memory 分层模型
2. recall search 处理 history callback
3. durable memory 与 active context 分离
4. 长会话下的 summary / recall helper 机制

### 6.4 不建议直接照搬的部分
1. Letta 的人格/长期 persona 叙事层
2. 让模型直接自由读写所有长期记忆的开放式模式
3. 对 MAXshot 当前阶段来说过重的 agent 自我演化流程

MAXshot 需要的是：
- 借鉴其 memory 分层
- 但保留我们自己的 Runtime Harness 和 guardrails

## 8. MAXshot Step 2 推荐架构（基于参考项目后的收敛）

### 8.1 职责拆分

#### 1. Session Resolver
- **Code 主导**
- 做确定性事情：
  - session/thread 归属
  - reset/fork 机制
  - TTL / lifecycle

#### 2. Turn Relation Classifier
- **LLM 主导**
- 做语义判断：
  - `continuation`
  - `clarification_reply`
  - `correction`
  - `new_topic_same_window`
  - `history_callback`

#### 3. Context Assembler
- **Code 主导**
- 读取：
  - session state
  - pending clarification
  - active capability registry
  - memory runtime refs

#### 4. Context Policy Engine
- **Code 主导**
- 决定：
  - 哪些上下文继承
  - 哪些清空
  - 哪些触发 recall

#### 5. Recall Adapter
- **Code + retrieval**
- 专门服务 `history_callback`
- 不直接混进一般 follow-up 逻辑

### 8.2 推荐责任边界

#### LLM 负责
- turn relation 语义判断
- `new topic` vs `continuation` vs `correction` 的高层理解

#### Code 负责
- session truth source
- registry load
- memory runtime assembly
- inheritance policy
- hard guardrails
- recall adapter 调度

### 8.3 不建议继续的旧方向
1. 不要继续把 Step 2 主要做成 regex / hardcoded follow-up 检测器
2. 不要把 `history_callback` 继续当成普通 follow-up 特例来修补
3. 不要让 Step 2 直接输出 intent/capability 结论
4. 不要把全部 memory 继续压缩成一层简单 ref list 而不分类型

## 10. 对后续开发的明确约束

### 10.1 后续 Step 2 开发必须遵守
1. Step 2 输出必须收敛为正式 `Context Envelope`
2. Step 2 不得继续承担 Step 3 的 intent/capability 职责
3. turn relation 不能再只靠规则函数作为主判断
4. history callback 必须作为正式分支设计，而不是补丁逻辑
5. memory runtime 后续升级时必须区分 memory 类型，而不是只加更多 ref

### 10.2 本文作为共享参考的用途
本文不是实现文档，而是：
- Step 2 架构借鉴的统一参考
- 后续技术讨论和设计评审的共享底稿
- 多项目共享时，用于解释“为什么 Step 2 要这么做”

## 11. 最终判断

对于 MAXshot 的 Step 2：

- **OpenClaw 解决了 session / thread / lifecycle / persistence 问题**
- **Letta 解决了 memory layering / recall / history callback 问题**
- **MAXshot 需要做的是把两者收敛到一个受控的 Conversation Context Manager 中**

换句话说：

- 不是照搬 OpenClaw
- 也不是照搬 Letta
- 而是：
  - 用 OpenClaw 解决“会话容器怎么建”
  - 用 Letta 解决“历史和记忆怎么分层”
  - 用 MAXshot 自己的 Harness 解决“LLM 与 Code 如何分工”

这才是 Step 2 的合理方向。
