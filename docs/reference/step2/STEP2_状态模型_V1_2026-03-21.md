# Step 2 状态模型 V1

- Status: Frozen

## 目的

定义 Step 2 所需的核心状态对象，避免后续继续沿用模糊的 `SessionBusinessContext`。

---

## 1. 当前问题

当前项目中的上下文状态偏向业务查询，字段大致围绕：
- scope
- chain
- protocol
- vault_name
- time_window_days
- aggregation

这不够支撑完整多轮对话管理，因为它缺：
- pending clarification
- relation-aware inheritance
- history callback 入口
- conversation-level summary

## 3. 设计原则

### 3.1 会话状态与业务状态分离
- 会话状态：session/thread/relation/policy
- 业务状态：scope/filter/time/aggregation

### 3.2 clarification 不能只是布尔值
必须保留：
- 原问题
- 缺槽
- 已问问题

### 3.3 memory 必须预留分层
即使第一版只实现 `ref_ids`，状态模型也应预留：
- `in_context`
- `recall_searchable`
- `durable_curated`

### 3.4 last summary 是必要字段
否则 history callback 会缺少结构化切入口。

---

## 4. 与当前实现的映射

### 当前可直接映射
- `scope` -> `active_scope`
- `chain/protocol/vault_name` -> `active_filters`
- `time_window_days` -> `active_time.time_window_days`
- `aggregation` -> `active_aggregation`

### 当前缺失
- `last_user_intent_summary`
- `last_answer_summary`
- `asked_question`
- `memory_layers`

---

## 5. 结论

Step 2 后续如果继续沿用 `SessionBusinessContext`，一定会再次失衡。  
后续编码应以本文定义的状态模型为准，再逐步映射到现有实现。
