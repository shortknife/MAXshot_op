## 4. 文件级详细迁移说明

### 4.1 `chat-request-preprocess.ts`

#### 当前角色
- 混合承担：
  - clarification context
  - follow-up context
  - parseIntent 前预处理

#### 目标角色
- 退化为 orchestration 壳

#### 迁移后应保留
- 调用链连接
- Step 1 输出到 Step 2 输入的装配
- Step 2 输出到 Step 3 输入的装配

#### 迁移后不应继续保留
- 具体的 Step 2 判断细节

---

### 4.2 `session-context.ts`

#### 当前角色
- session business context store
- continuation/follow-up heuristics

#### 目标角色
- 一分为二：
  1. store 能力保留，迁到 `session-resolver` / `context-assembler`
  2. heuristics 退化为 `turn-relation-classifier` fallback

#### 需要保留的资产
- 现有短期上下文存取能力
- TTL 逻辑

#### 需要替换的部分
- 业务导向的 relation 判断逻辑

---

### 4.3 `query-clarification.ts`

#### 当前角色
- pending clarification store
- clarification 消费逻辑
- 一部分“新问题 vs 旧补充”判断

#### 目标角色
- clarification store 作为状态来源保留
- “如何消费 clarification”交给 `context-policy`
- “是不是新问题”交给 `turn-relation-classifier`

#### 需要保留
- clarification 状态存取

#### 需要迁出
- relation 判断逻辑
- query 拼接式补丁逻辑

---

### 4.4 `capability-catalog.ts`

#### 当前角色
- active capability registry 装载

#### 目标角色
- 继续作为 registry source
- 被 `context-assembler` 调用

#### 结论
- 基本不需要大改
- 这是 Step 2 当前较稳定的部分

---

### 4.5 `memory-refs.ts`

#### 当前角色
- 简化版 memory runtime refs

#### 目标角色
- 继续作为 memory runtime 构建基础
- 后续被：
  - `context-assembler`
  - `recall-adapter`
 共同消费

#### 需要补的不是迁移，而是升级
- 预留 memory layers
