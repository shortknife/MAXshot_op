## 1. 目标

本计划只解决一件事：

**把当前“规则驱动的业务查询续接 MVP”升级为正式的 `Conversation Context Manager`。**

Step 2 不做：
- capability 匹配
- intent 最终分类
- gate
- router
- capability 执行

Step 2 只做：
1. `session_resolution`
2. `turn_relation`
3. `conversation_context`
4. `registry_context`
5. `memory_runtime`
6. `context_policy`

最终输出统一 `Context Envelope` 给 Step 3。

---

## 2. 落地原则

### 2.1 责任分配

#### LLM 主导
- `turn_relation` 判断
- `continuation / clarification_reply / correction / new_topic_same_window / history_callback` 语义分类

#### Code 主导
- session/thread 装载
- active capability registry 装载
- memory runtime 装载
- context inheritance / discard policy
- hard guardrails
- recall adapter 调度

### 2.2 不直接推翻现有主链

采用 **兼容式替换**：
- 先把 Step 2 从现有 `prepareChatRequest()` 中抽离为独立层
- 再逐步把旧规则函数退化为 fallback
- 最后由 Step 2 输出正式 `Context Envelope`

### 2.3 先收口，再扩功能

本轮不扩：
- 新 capability
- 新入口 channel
- 新 memory 存储后端

本轮只做：
- 结构收口
- 责任重排
- 上下文管理能力从规则补丁升级为正式模块

---

## Phase B - 正式建立 Session Resolver

### 目标
把“当前消息属于哪个 session/thread”从隐含逻辑改成正式输出。

### 产出
新增模块：
- `admin-os/lib/chat/context-manager/session-resolver.ts`

输出：
- `session_resolution.session_id`
- `session_resolution.thread_action`
- `session_resolution.confidence`

### Code 责任
1. 读取 Step 1 的 `Entry Envelope`
2. 读取现有 session store
3. 处理确定性条件：
   - 显式 session_id
   - session TTL
   - reset/fork 触发条件
   - pending clarification 是否存在

### 当前可复用部分
- `session-context.ts` 的存取能力
- `query-clarification.ts` 的 pending clarification store

### 本阶段不做
- 不让 LLM 决定 session_id
- 不做 capability 判断

### 验收
1. 任意输入都有明确 `session_resolution`
2. `thread_action` 至少支持：
   - `continue`
   - `fork_new`
   - `reset`
3. 输出标准化 JSON

### 备注
这里要借鉴 OpenClaw 的点：
- session 是正式对象
- session truth source 集中
- thread/topic 是一级边界

## Phase C - 用 LLM 建立 Turn Relation Classifier

### 目标
把当前规则函数主导的 follow-up/new topic 判断，升级成 LLM 主导的语义分类器。

### 产出
新增模块：
- `admin-os/lib/chat/context-manager/turn-relation-classifier.ts`

新增 Prompt：
- `turn_relation_classifier_op_v1`

输出 contract：

```json
{
  "type": "continuation",
  "confidence": 0.92,
  "reason": "..."
}
```

### 必须支持的 relation taxonomy
- `new_session`
- `continuation`
- `clarification_reply`
- `correction`
- `new_topic_same_window`
- `history_callback`

### LLM 输入建议
输入必须受控，只给：
1. 当前 `raw_query`
2. 最近有限轮对话摘要/关键 turn
3. pending clarification 状态
4. active conversation context 摘要

不要直接喂整段杂乱 session。

### Code 仍要保留的 guardrails
1. 显式 `/new` 或 reset 类命令 -> 直接 `new_session`
2. 空 query / 非法输入 -> 直接 reject
3. 低置信度时可 fallback 到保守策略

### 当前旧逻辑如何处理
这些函数退化为 fallback，而不是主判断：
- `looksLikeFollowUp`
- `looksLikeOptionReply`
- `looksLikeStandaloneQuestion`
- `inferBusinessScope`

### 验收
重点测 6 类：
1. `最近7天` -> `clarification_reply`
2. `那 arbitrum 呢？` -> `continuation`
3. `不对，我问的是 vault 不是 market` -> `correction`
4. `你能描述什么是MAXshot么？` -> `new_topic_same_window`
5. `我之前问的 dForce 那个再看一下` -> `history_callback`
6. 明确 reset -> `new_session`

## Phase D - 建立 Context Assembler

### 目标
把 session state、pending clarification、registry、memory refs 组装成统一 `Context Envelope`。

### 产出
新增模块：
- `admin-os/lib/chat/context-manager/context-assembler.ts`

组装内容：
1. `conversation_context`
2. `registry_context`
3. `memory_runtime`
4. `effective_query`
5. `context_ready`

### Code 责任
1. 读取 session state
2. 读取 pending clarification
3. 读取 active capability registry
4. 构建 memory runtime
5. 封成标准 envelope

### 当前可复用部分
- `capability-catalog.ts`
- `memory-refs.ts`
- `session-context.ts`
- `query-clarification.ts`

### 当前需要纠正的点
1. `SessionBusinessContext` 过于 business-only
2. 需要升级成 `ConversationContextState`

建议至少扩成：
- `active_scope`
- `pending_clarification`
- `active_filters`
- `active_time_range`
- `active_aggregation`
- `last_user_intent_summary`
- `last_answer_summary`

### 验收
1. Step 2 输出不再是散落字段
2. 必须有统一 `Context Envelope`
3. Step 3 只消费这个 envelope，不再自己拼上下文

---

## Phase E - 建立 Context Policy Engine

### 目标
根据 `turn_relation` 决定哪些上下文继承、清空、覆盖或触发回溯。

### 产出
新增模块：
- `admin-os/lib/chat/context-manager/context-policy.ts`

### 策略建议

#### `continuation`
- 继承 active business context
- 允许缩小范围
- 不清 session

#### `clarification_reply`
- 只补 pending clarification 的缺槽
- 不允许旧 context 覆盖新明确值

#### `correction`
- 继承 session
- 保留一部分上下文
- 覆盖被用户纠正的字段

#### `new_topic_same_window`
- 保留 session 元信息
- 清空业务上下文
- 清空 pending clarification

#### `history_callback`
- 保留 session
- 不直接沿用旧业务 query 文本
- 转交 Recall Adapter

#### `new_session`
- 创建新的会话上下文
- 不继承旧业务上下文

### 验收
1. 新问题不被旧 clarification 吞掉
2. correction 不再当普通 follow-up
3. new topic 不再错误继承 APY / vault / chain 上下文

---

## Phase F - 建立 Recall Adapter

### 目标
为 `history_callback` 建立正式分支，不再把它当 follow-up 补丁。

### 产出
新增模块：
- `admin-os/lib/chat/context-manager/recall-adapter.ts`

### 第一版不要做太重
先做轻量版：
1. 基于 session history / prior turns 做结构化检索
2. 输出：
   - `recall_hits`
   - `recall_summary`
   - `recall_confidence`

### 第二版再考虑
- 分层 recall
- durable memory
- vector / semantic retrieval

### 参考来源
这里主要借鉴 Letta：
- `conversation_search`
- `archival_memory_search`
- recall / archival 分层思想

### 验收
至少支持：
- `我之前问的 dForce 那个再看一下`
- `回到刚才那个 vault`

而不是把 query 文本硬拼到当前输入上。

## Phase G - 接回主链并退役旧逻辑

### 目标
让 Step 2 成为正式主链，而不是旁路补丁。

### 动作
1. `prepareChatRequest()` 改为：
   - 接收 Step 1 Envelope
   - 调用 Step 2 Context Manager
   - 再进入 Step 3

2. 旧逻辑退化：
   - 旧 follow-up heuristics -> fallback only
   - 旧 clarification 拼接逻辑 -> policy-controlled only

3. Step 3 只接：
   - `effective_query`
   - `conversation_context`
   - `registry_context`
   - `memory_runtime`

### 验收
1. Step 2 可独立测
2. Step 3 不再自己推断 session/continuation/new topic
3. 主链通过现有 regression

## 7. 推荐实施顺序

严格按这个顺序：

1. **Phase A**
   - 抽离 Step 2 边界
2. **Phase B**
   - Session Resolver
3. **Phase C**
   - Turn Relation Classifier
4. **Phase D**
   - Context Assembler
5. **Phase E**
   - Context Policy Engine
6. **Phase F**
   - Recall Adapter
7. **Phase G**
   - 接回主链，退役旧逻辑

原因：
- 先把结构拉开
- 再替换语义判断
- 最后才接回主链

这样风险最小。
