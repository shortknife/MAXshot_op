# MVP Product Brief (v0.2 Product-Oriented)

## 1) 产品一句话
MAXshot MVP 是一个面向小团队的增长助手：
- **运营查询引擎（Ops Copilot）**：自然语言问数，快速拿到可读结论。
- **内容发布引擎（Marketing Copilot）**：自然语言产出可发布文案，并支持快速改写。

---

## 2) 目标用户
1. **一线运营/产品人员**：要“查得准”，不想看 SQL / JSON。
2. **市场/内容人员**：要“发得快”，从主题到草稿一次完成。
3. **管理员（Admin）**：要“可控可审计”，对敏感动作有人审。

---

## 3) MVP 要解决的核心问题
1. 查询慢、解释难：业务问题需要跨系统找数据。
2. 发帖慢、改稿慢：从想法到发布链路过长。
3. 风险不可见：执行与写入缺少清晰审计。

---

## 4) MVP 核心能力（麻雀虽小，五脏俱全）

## A. 用户侧（必须可直接使用）
1. **自然语言查询**（Ops）
- 输入业务问题（如“最近 7 天失败执行有哪些原因？”）
- 输出：人话结论 + 关键数据点 + 可追溯来源

2. **自然语言内容生成**（Marketing）
- 输入目标（如“写一条促活帖，语气专业，含 CTA”）
- 输出：可用草稿 + 2 个改写动作（更短 / 更有行动号召）

## B. 管理侧（必须可治理）
1. **审计链路可读**（Entry -> Confirm -> Run -> Outcome -> Audit）
2. **高风险动作人工闸门**（Confirm/Replay/Retry/Expire 等）

---

## 5) MVP 不做什么（边界）
- 不做全自动写入与自动发布
- 不做企业级 SSO / IAM 全量上线
- 不做多入口全量生产化（TG / Notion 先留接口）

---

## 6) 成功标准（产品口径）

## 用户价值指标
1. 运营问题：用户可在 1 次对话中拿到可读答案（不需要看 JSON）
2. 内容产出：用户可在 1 次输入后拿到可发布草稿

## 系统可信指标
1. 关键执行链路可审计、可回放
2. 失败可解释（明确原因 + 下一步建议）

---

## 7) 典型用户旅程
1. 用户输入自然语言问题/需求
2. 系统返回可读结果（查询结论或内容草稿）
3. 管理员在后台查看执行与审计（必要时人工确认）
4. 用户继续迭代提问或改稿

---

## 8) 当前版本差距（下一步）
1. 用户入口需与 Admin 入口分层（隐藏 operator/token/json）
2. 错误返回需全面人话化（如 `missing_topic` -> “请补充主题”）
3. Marketing 侧需增加“改写/定风格/定渠道”快捷操作
