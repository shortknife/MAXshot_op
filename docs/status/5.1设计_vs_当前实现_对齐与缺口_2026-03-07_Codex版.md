## 二、入口层（Entry Layer）

| v5.1 / v5.0 设计 | 当前实现 | 对齐/缺口 |
|------------------|----------|-----------|
| 三入口：TG / Admin OS / Notion / System | Admin OS 已完整可测；TG / Notion 仍以参考实现和接口为主，不是当前用户主链路 | ⚠️ **部分**：Admin OS 对齐较多，TG/Notion 未形成当前统一产品入口 |
| raw_query 必经 Intent Analyzer | **Admin OS Chat** 已走 `/api/chat/ask -> parseIntent -> clarification/business_query`；TG 主入口未验证为同一链路 | ⚠️ **部分**：Chat 主链路对齐；三入口统一未完成 |
| 统一 EntryRequest | execution/run、ops、marketing 有 envelope；Chat 路径仍有专用输入体 | ⚠️ **部分** |
| Continue Chat / 最多多轮澄清 | Chat 已有 session + clarification + follow-up；当前默认允许最多 3 轮逻辑 | ✅ **在 Chat 主链路已实现** |
| Side Effect 先 pending_confirmation | operations / confirmations / audit 已有 `pending_confirmation`、confirm / reject / expire / replay | ✅ |

**更新结论**：相对于 2026-02 文档，`Admin OS Chat` 这条 raw_query 主路径已经明显前进；但“三入口统一”仍未完成。

## 三、Intent 与 Gate

| v5.1 / v5.0 设计 | 当前实现 | 对齐/缺口 |
|------------------|----------|-----------|
| raw_query -> Intent Analyzer（LLM）输出 intent/slots | `lib/intent-analyzer/deepseek-client.ts` 已接入；`/api/chat/ask` 使用它做 business_query / out_of_scope / clarification | ✅ **Chat 主链路已实现** |
| Gate 校验能否负责执行，不轻易 reject | Chat 中已有 `out_of_scope`、`missing_required_clarification`、`insufficient_business_data` 等标准化出口 | ✅ **部分对齐** |
| Entry Gate / Capability Gate 双层角色 | 代码里已有入口判断和 capability 内判断，但还没完全显式拆成两个统一模块 | ⚠️ **部分** |
| Intent 严禁输出 capability_chain | 当前 intent 输出只落 `intent + extracted_slots`，未见 capability_chain 直接进入 Router | ✅ |

**更新结论**：Intent 这部分比 2026-02 状态更好，尤其在 Chat 主链路上已经从“固定规则”升级为“LLM + 规则兜底”；但 Gate 仍未完整平台化。

## 四、Router 与 Capability

| v5.1 / v5.0 设计 | 当前实现 | 对齐/缺口 |
|------------------|----------|-----------|
| Router = 确定性调度器 | `router-main.ts`、`resolveCapability`、audit 路径存在；execution/action 仍是确定性映射 | ✅ |
| Router 只认 Execution ID | operations / evolution / marketing 这条链路较对齐；Chat 主链路仍是直接 `chat/ask`，未统一先封印成 execution | ⚠️ **部分** |
| Capability 受 Registry allowlist 调用 | capability index / registry 雏形存在；主链路仍有一些直调实现 | ⚠️ **部分** |
| Capability 不拥有查询主权 | 设计上开始收口，但 `business-data-query.ts` 目前仍混有 source 选择、summary、next_actions、部分语义默认 | ❌ **缺口** |

**更新结论**：Router 核心思想还在，但 `business_data_query` 这个能力目前仍然“太厚”，还不是 FSD 期望的纯 runtime capability。

## 十一、当前缺口汇总（按优先级）

### P0 缺口（当前最该补）

1. `business_data_query` 从“厚能力”拆成：
   - semantic index / planner
   - runtime provider
   - normalizer / output
2. `最近7天 APY` 默认语义冻结为：`最近7天平均 APY`
3. `memory_refs` 真正接入 business query 主链路
4. `next_actions` 完全降级为 follow-up examples，不再携带“按钮流程”含义

### P1 缺口

5. Chat 主链路与 execution/sealer 统一
6. 三入口（TG / Admin OS / Notion）按同一 EntryRequest 体系收口
7. Output 统一成 `exit_type` 主导而非分页面散落处理

### P2 缺口

8. Capability Registry / Semantic Index 真正驱动 Router
9. provider fallback / attempt ladder / contract gate 全面平台化

## 十二、最终结论

与 `2026-02` 版本相比，当前进展主要体现在：

- Chat 主链路已经真正具备 `Intent + Clarification + Business Query`
- Ops / Marketing / Evolution / Audit 四块 MVP 基本都能跑
- Vanna 子进程路径、business schema builder、semantic index 文档已补齐

但从 v5.1 / FSD 的严格定义看，当前仍然处于：

- **功能上已进入 MVP 可测 / 可演示状态**
- **架构上仍处于“半平台化、半定制实现”阶段**

因此本版判断为：

- **MVP 功能完成度：80%+**
- **FSD 平台化对齐度：约 68%**
